   1: # Aircraft Identity And 3D Models
   2: 
   3: ## Context
   4: 
   5: Worldviewer already shows live aircraft from OpenSky in the browser. Arthur wants three upgrades:
   6: 
   7: 1. better aircraft identity
   8: 2. better aircraft type information
   9: 3. a more representative visual, including 3D models at altitude
  10: 
  11: The design should stay small and should preserve the current GitHub Pages story. That means:
  12: 
  13: - no new aircraft backend
  14: - no feature flags
  15: - no migration choreography
  16: - no second live aircraft pipeline
  17: 
  18: ## What Matters
  19: 
  20: - Aircraft must keep working on the static GitHub Pages build.
  21: - The design must stay honest about what the data really tells us.
  22: - The map must stay responsive in a normal browser.
  23: - Each step must be independently useful.
  24: 
  25: ## Non-Goals
  26: 
  27: - Perfect airline-grade flight identity
  28: - Full photorealistic aircraft simulation
  29: - Per-aircraft animation histories or contrails
  30: - A large aircraft data platform
  31: - A new backend just for aircraft metadata
  32: 
  33: ## Hard Constraints
  34: 
  35: OpenSky live state vectors give us callsign, position, speed, heading, altitude, and optional aircraft category. They do not directly give us a trustworthy commercial flight number or exact airframe model for every aircraft.
  36: 
  37: So:
  38: 
  39: - `callsign` is real and available now
  40: - `flight number` is best-effort only unless another live source is added later
  41: - `aircraft category` can be requested with `extended=1`
  42: - `exact aircraft type` needs a second lookup keyed by `icao24`
  43: 
  44: Also:
  45: 
  46: - the existing map pipeline is built around one shared `LiveTrack` shape
  47: - Step 3 must not dump a large 3D library into the main bundle
  48: - Step 3 must prove it is worth shipping before we commit to it
  49: 
  50: ## Decision Summary
  51: 
  52: Implement this in three steps:
  53: 
  54: ### Step 1
  55: 
  56: Enrich the existing OpenSky path with category, better labels, and better 2D aircraft visuals.
  57: 
  58: ### Step 2
  59: 
  60: Add a static aircraft identity lookup keyed by `icao24` so we can attach type code, registration, manufacturer, and model without adding a new runtime service.
  61: 
  62: ### Step 3
  63: 
  64: Attempt a limited 3D aircraft layer, but only after a small go/no-go spike proves that the assets, coverage, and rendering cost are acceptable.
  65: 
  66: ## Shared Data Model
  67: 
  68: Do not introduce a separate `AircraftTrack` pipeline.
  69: 
  70: Keep one shared `LiveTrack` type and add aircraft-only optional fields to it. Ships will leave those fields unset.
  71: 
  72: That avoids forking:
  73: 
  74: - `SnapshotMessage`
  75: - `TrafficClient`
  76: - `tracksToGeoJSON`
  77: - popup rendering
  78: - traffic layer updates
  79: 
  80: The shape becomes:
  81: 
  82: ```ts
  83: type LiveTrack = {
  84:   id: string;
  85:   kind: "aircraft" | "ship";
  86:   lng: number;
  87:   lat: number;
  88:   heading: number | null;
  89:   speedKnots: number | null;
  90:   altitudeMeters: number | null;
  91:   label: string | null;
  92:   source: "opensky" | "aisstream";
  93:   updatedAt: number;
  94: 
  95:   // aircraft-only optional fields
  96:   onGround?: boolean | null;
  97:   callsign?: string | null;
  98:   flightCode?: string | null;
  99:   aircraftCategory?: number | null;
 100:   geoAltitudeMeters?: number | null;
 101:   aircraftTypeCode?: string | null;
 102:   registration?: string | null;
 103:   manufacturer?: string | null;
 104:   model?: string | null;
 105:   renderModelKey?: string | null;
 106: };
 107: ```

 109: ## Step 1: Richer Live Aircraft Metadata
 110: 
 111: ### Goal
 112: 
 113: Make the current aircraft points more informative without changing the architecture.
 114: 
 115: ### Changes
 116: 
 117: - request OpenSky states with `extended=1`
 118: - parse and store:
 119:   - `callsign`
 120:   - `aircraftCategory`
 121:   - `geoAltitudeMeters`
 122:   - existing heading, speed, and barometric altitude
 123: - replace the current `^` glyph with a small aircraft silhouette set keyed by coarse class
 124: - add a small callsign formatter for best-effort flight display
 125: 
 126: ### OpenSky Field Map
 127: 
 128: The implementation must use named constants for the state-vector indices.
 129: 
 130: For the fields this feature needs:
 131: 
 132: - `0`: `icao24`
 133: - `1`: `callsign`
 134: - `5`: longitude
 135: - `6`: latitude
 136: - `7`: barometric altitude
 137: - `9`: velocity
 138: - `10`: true track
 139: - `13`: geometric altitude
 140: - `17`: category when `extended=1`
 141: 
 142: ### Browser Feasibility Note
 143: 
 144: As of `2026-03-12`, anonymous browser requests to `states/all?extended=1` were manually verified to return 18-field state vectors with CORS enabled.
 145: 
 146: If OpenSky stops returning category data for anonymous callers later, Step 1 still works:
 147: 
 148: - callsign stays available
 149: - silhouettes fall back to a generic aircraft symbol
 150: 
 151: The design does not depend on category being perfect.
 152: 
 153: ### Flight Code Rule
 154: 
 155: Do not pretend to infer a true commercial flight number.
 156: 
 157: Only derive `flightCode` when the callsign matches this exact rule:
 158: 
 159: ```ts
 160: /^([A-Z]{3})(\d{1,4}[A-Z]?)$/
 161: ```
 162: 
 163: If it does not match, show only the raw callsign.
 164: 
 165: ### Rendering
 166: 
 167: - low zoom: clusters as today
 168: - mid zoom: 2D aircraft silhouette instead of `^`
 169: - popup fields:
 170:   - `flightCode` if present
 171:   - raw `callsign`
 172:   - category label
 173:   - speed
 174:   - altitude
 175:   - age
 176: 
 177: ### Why This First
 178: 
 179: This step gives visible value immediately and needs no new infrastructure.
 180: 
 181: ## Step 2: Static Aircraft Identity Lookup
 182: 
 183: ### Goal
 184: 
 185: Attach exact aircraft type information where possible without adding a new runtime dependency.
 186: 
 187: ### Source
 188: 
 189: Use the public downloadable OpenSky aircraft database CSV as the only metadata source in scope for this step.
 190: 
 191: Do not scrape third-party aircraft sites.
 192: 
 193: The README and credits should cite OpenSky for both live traffic and aircraft metadata.
 194: 
 195: ### Refresh Policy
 196: 
 197: Keep this manual and boring:
 198: 
 199: - download a recent CSV snapshot when we want to refresh
 200: - run one local build script
 201: - commit the generated compact output
 202: 
 203: Quarterly refresh is enough for this personal app. We do not need an automated ingestion pipeline.
 204: 
 205: ### Stored Fields
 206: 
 207: Retain only:
 208: 
 209: - `icao24`
 210: - `registration`
 211: - `typeCode`
 212: - `manufacturer`
 213: - `model`
 214: 
 215: Drop everything else during generation.
 216: 
 217: ### Static Artifact Strategy
 218: 
 219: Use 256 static JSON shards by first two hex characters:
 220: 
 221: ```text
 222: public/
 223:   aircraft-identity/
 224:     00.json
 225:     01.json
 226:     ...
 227:     fe.json
 228:     ff.json
 229: ```
 230: 
 231: Why 256:
 232: 
 233: - still mechanically simple and deterministic
 234: - keeps the real OpenSky skew inside the shard-size budget in practice
 235: - still static-host friendly for a personal app
 236: 
 237: ### Generation Guardrails
 238: 
 239: The generation script must log output sizes for every shard before anything is committed.
 240: 
 241: If any shard is obviously too large for this app, stop and revise the shard scheme before shipping the refresh. A good warning threshold is:
 242: 
 243: - about `5 MB` uncompressed for a single shard, or
 244: - about `1.5 MB` gzip for a single shard
 245: 
 246: We do not need automatic repartitioning. Manual review is enough.
 247: 
 248: ### Runtime Behaviour
 249: 
 250: - when an aircraft becomes visible, look at the first two hex characters of `icao24`
 251: - fetch that shard if it is not already in the session cache
 252: - merge identity fields into matching visible aircraft
 253: - if a fetch fails or an identity is missing, keep Step 1 behaviour
 254: 
 255: The cache is session-scoped only:
 256: 
 257: ```ts
 258: Map<string, Record<string, AircraftIdentity>>
 259: ```
 260: 
 261: No local storage. No persistence.
 262: 
 263: ### Flight Number Handling
 264: 
 265: Step 2 still does not claim to know a true commercial flight number.
 266: 
 267: It only improves:
 268: 
 269: - registration
 270: - type code
 271: - manufacturer
 272: - model
 273: 
 274: ### Rendering Impact
 275: 
 276: This step unlocks better visual selection:
 277: 
 278: - `B738` -> 737-family model key
 279: - `A20N` -> A320-family model key
 280: - `B77W` -> 777-family model key
 281: - fallback to category class when type code is missing
 282: 
 283: ## Step 3: 3D Aircraft At Altitude
 284: 
 285: ### Goal
 286: 
 287: Render aircraft as actual 3D objects at altitude when the camera is close enough for that to matter.
 288: 
 289: ### Go / No-Go Spike First
 290: 
 291: Step 3 started with a narrow proof-of-concept before the full build.
 292: 
 293: The spike answered three questions:
 294: 
 295: 1. can we render one aircraft model at live altitude inside a MapLibre custom layer without breaking map interaction?
 296: 2. do we have at least six redistribution-safe class shapes that look good enough for a personal app?
 297: 3. do representative views contain enough aircraft with usable altitude data to make the feature visible?
 298: 
 299: The practical outcome was:
 300: 
 301: - yes, the custom-layer path is viable
 302: - yes, external model assets are unnecessary; small procedural class meshes are good enough
 303: - yes, `geoAltitudeMeters` plus barometric fallback is enough to make the layer visibly useful
 304: 
 305: So the shipped Step 3 uses bounded procedural class meshes instead of a glTF asset pack.
 306: 
 307: ### Rendering Technology
 308: 
 309: Use a MapLibre custom 3D layer.
 310: 
 311: Load three.js dynamically the first time 3D aircraft are actually needed. Do not ship it in the initial application bundle.
 312: 
 313: Until the lazy-loaded 3D path is ready, keep showing 2D aircraft symbols. Do not show an empty transition state.
 314: 
 315: ### Why Not A Bigger 3D System
 316: 
 317: We do not need:
 318: 
 319: - a second renderer
 320: - a persistent aircraft scene graph across the whole world
 321: - a large asset pipeline
 322: - dozens of type-specific models up front
 323: 
 324: We only need a bounded custom layer that renders a small set of visible aircraft.
 325: 
 326: ### Model Strategy
 327: 
 328: Start with six class models only:
 329: 
 330: - `narrow-body`
 331: - `wide-body`
 332: - `regional-jet`
 333: - `bizjet`
 334: - `prop`
 335: - `helicopter`
 336: 
 337: Do not make Step 3 depend on type-specific `.glb` assets existing.
 338: 
 339: For the initial build, implement those six classes as small procedural meshes built from Three.js primitives. That keeps the redistribution story trivial and avoids an asset pipeline entirely.
 340: 
 341: If later we find license-compatible, low-poly models for common exact types, we can add overrides on top:
 342: 
 343: - `B738`
 344: - `A320`
 345: - `A21N`
 346: - `B77W`
 347: - `B788`
 348: 
 349: But class models are the required path. Exact overrides are optional polish.
 350: 
 351: ### Asset Rule
 352: 
 353: Only use redistribution-safe geometry.
 354: 
 355: For the initial build, procedural class meshes satisfy that rule, so no external model pack is required. If we later add exact-type overrides, those assets must still be free and redistribution-safe.
 356: 
 357: ### Activation Rules
 358: 
 359: Use hysteresis so the view does not flicker between 2D and 3D.
 360: 
 361: Activate 3D only when all are true:
 362: 
 363: - zoom is at least `10.5`
 364: - pitch is at least `45`
 365: - visible renderable airborne aircraft count is at most `24`
 366: 
 367: Drop back to 2D when any are true:
 368: 
 369: - zoom falls below `10`
 370: - pitch falls below `35`
 371: - visible renderable airborne aircraft count rises to `32` or more
 372: 
 373: The visible-count check must be based on the raw aircraft track array filtered by viewport bounds and 3D renderability, not rendered map features. That avoids cluster interference while keeping grounded / non-renderable aircraft on the 2D path.
 374: 
 375: ### Altitude Policy
 376: 
 377: Step 3 should use:
 378: 
 379: - `geoAltitudeMeters` when present
 380: - otherwise `altitudeMeters` as a visual fallback only
 381: 
 382: At the zoom and pitch where Step 3 activates, the visual error from barometric fallback is acceptable for a representative personal-app view.
 383: 
 384: The popup should make the altitude source explicit when needed.
 385: 
 386: ### Positioning
 387: 
 388: - convert the aircraft position with `MercatorCoordinate.fromLngLat(lngLat, chosenAltitude)`
 389: - rotate by heading / true track
 390: - keep on-ground aircraft on the 2D symbol path
 391: 
 392: ### Performance Budget
 393: 
 394: Step 3 is only acceptable if it stays cheap enough to matter.
 395: 
 396: Acceptance target:
 397: 
 398: - keep interactive frame rate comfortably interactive in representative high-pitch views
 399: 
 400: Primary guardrails for this app:
 401: 
 402: - bounded aircraft count (`24` on, `32` off)
 403: - lazy-loaded Three.js path
 404: - simple procedural class meshes
 405: - automatic fallback to 2D outside the activation window
 406: 
 407: If that target is missed:
 408: 
 409: - lower the cap
 410: - raise the zoom threshold
 411: - reduce model complexity
 412: 
 413: Do not keep the feature on by default if it clearly harms navigation.
 414: 
 415: ## UI Changes
 416: 
 417: Extend the aircraft popup to prefer this order:
 418: 
 419: 1. best-effort `flightCode`
 420: 2. raw `callsign`
 421: 3. registration
 422: 4. manufacturer + model
 423: 5. type code
 424: 6. category label
 425: 
 426: Do not add a new global toggle for Step 1 or Step 2.
 427: 
 428: For Step 3, keep automatic activation by camera state in the first pass. If that proves confusing in practice, a single local `3D Aircraft` checkbox is an acceptable follow-up, but it is not part of the initial design.
 429: 
 430: ## Validation Plan
 431: 
 432: ### Step 1
 433: 
 434: - unit tests for OpenSky parsing with `extended=1`
 435: - unit tests for callsign-to-flight-code formatting
 436: - manual check that GitHub Pages still shows aircraft
 437: 
 438: ### Step 2
 439: 
 440: - unit tests for CSV-to-shard generation
 441: - unit tests for shard load and merge logic
 442: - manual check for missing-identity fallback
 443: - manual check that shard fetch failure is non-blocking
 444: - manual review of shard-size logs before committing a refresh
 445: 
 446: ### Step 3 Spike
 447: 
 448: - one live aircraft rendered at altitude in a custom layer
 449: - one lazy-load proof that 3D code is not in the initial path
 450: - one concrete class-model strategy confirmed
 451: - one bounded activation profile that keeps the map responsive
 452: 
 453: ### Step 3 Full Build
 454: 
 455: - unit tests for model key selection
 456: - unit tests for controller fallback and success-path loading behaviour
 457: - manual browser checks for altitude placement, heading rotation, and LOD cutoff when convenient
 458: - manual check that 2D aircraft remain visible until the 3D path finishes loading when convenient
 459: 
 460: ## Risks
 461: 
 462: - callsign is not the same as a true airline flight number
 463: - OpenSky category data could change or disappear for anonymous callers later
 464: - OpenSky aircraft metadata is crowdsourced and can be stale or incomplete
 465: - even 256 shards can still be uneven by prefix
 466: - 3D aircraft can get expensive fast if thresholds are too loose
 467: - procedural class meshes are intentionally approximate, not exact airframes
 468: 
 469: ## Recommendation
 470: 
 471: Proceed exactly in order:
 472: 
 473: 1. richer metadata and 2D silhouettes
 474: 2. static `icao24` identity lookup
 475: 3. Step 3 bounded procedural layer once the proof-of-concept confirms it is worth shipping
 476: 
 477: That keeps the design honest, keeps GitHub Pages viable, and avoids turning a personal app into an aviation platform.
 478:

=== README.md ===
   1: # Worldviewer
   2: 
   3: Browser-based Earth twin built with MapLibre GL JS. It starts in orbit, supports globe projection, terrain, 3D buildings, location search, and cinematic fly-to jumps into street-scale views.
   4: 
   5: ## Run
   6: 
   7: ```powershell
   8: npm install
   9: npm run dev
  10: ```
  11: 
  12: `npm run dev` now starts both Vite and the local traffic relay. If you only want the frontend dev server, use `npm run dev:web`.
  13: 
  14: ## Aircraft Metadata Refresh
  15: 
  16: Aircraft identity shards are generated from the public OpenSky aircraft metadata snapshots. Download a complete monthly CSV into a local scratch folder, then run:
  17: 
  18: ```powershell
  19: npm run refresh:aircraft-identity -- --input tmp/aircraft-database-complete-2025-08.csv
  20: ```
  21: 
  22: This writes 256 JSON shards under `public/aircraft-identity/`, keyed by the first two hex characters of `icao24` (`00.json` .. `ff.json`), and logs each shard's raw and gzip size so the guardrails are visible before commit.
  23: 
  24: ## Configuration
  25: 
  26: - `AISSTREAM_API_KEY`: optional, enables the live ship layer. If it is missing, aircraft still work and ships stay unavailable in the UI.
  27: 
  28: ## Stack
  29: 
  30: - MapLibre GL JS for globe rendering and camera control
  31: - OpenFreeMap vector tiles and labels
  32: - EOX Sentinel-2 Cloudless imagery for realistic satellite texture
  33: - AWS Terrain Tiles / Terrarium DEM for terrain and relief
  34: - OpenStreetMap Nominatim search for lightweight geocoding
  35: - OpenSky direct from the browser for live aircraft
  36: - OpenSky aircraft metadata snapshots for static aircraft identity shards
  37: - AISStream via the local relay for live ships
  38: 
  39: ## Notes
  40: 
  41: - This is the realistic open-data version of the feature. It reaches street-scale navigation with terrain, roads, labels, and extruded buildings, but it is not worldwide photogrammetry.
  42: - Live traffic is intended for personal, non-commercial use and depends on public community feeds. Coverage and freshness vary by region and provider.
  43: - GitHub Pages can host the static globe build and live aircraft, because live aircraft and static aircraft identity both come from OpenSky-compatible browser/static paths. Live ships still need a relay, so the ship toggle stays unavailable on the Pages build.
  44: - At higher zoom and pitch, airborne aircraft switch from 2D symbols to bounded 3D class models in the browser. This only activates when the visible airborne aircraft count stays low enough to keep the map responsive.
  45: - The external services above are public community/demo services. For sustained production traffic, swap them for self-hosted or contracted equivalents.
  46: 
  47: ## Credits
  48: 
  49: - MapLibre GL JS
  50: - OpenFreeMap
  51: - EOX Maps Sentinel-2 Cloudless
  52: - AWS Terrain Tiles / Terrarium
  53: - OpenStreetMap Nominatim
  54: - OpenSky Network for live aircraft state vectors and aircraft metadata snapshots
  55: - AISStream for live ship traffic

 648: function wireMap(mapInstance: Map): void {

 662:   const markSceneReady = () => {
 663:     if (sceneReady) {
 664:       return;
 665:     }
 666: 
 667:     sceneReady = true;
 668:     mapInstance.resize();
 669:     syncViewState(mapInstance);
 670:     statusPill.textContent = mapState.stressModeActive

 683:

 685:   mapInstance.on("load", () => {
 686:     markSceneReady();
 687:   });
 688: 
 689:   mapInstance.on("idle", () => {
 690:     if (statusPill.classList.contains("is-error")) {
 691:       return;

 693:     statusPill.textContent = mapState.stressModeActive
 694:       ? "Performance mode active for dense street detail."

 703:     mapState.userInteracting = false;
 704:     scheduleViewSync();

 706:   });
 707: 
 708:   mapInstance.on("move", () => {
 709:     scheduleViewSync();
 710:   });

 712:   mapInstance.on("click", (event) => {
 713:     const feature = mapInstance
 714:       .queryRenderedFeatures(event.point, { layers: [...FEATURE_QUERY_LAYERS] })
 715:       .find(Boolean);
 716:

 718:       activePopup?.remove();
 719:       activePopup = null;
 720:       return;
 721:     }
 722:

 724:       pickString(feature.properties?.name) ??
 725:       pickString(feature.properties?.["name_en"]) ??
 726:       prettifyLayerName(feature.layer.id);
 727:     const secondary = pickString(feature.properties?.class) ?? pickString(feature.properties?.type);
 728:     const height = pickString(feature.properties?.render_height) ?? pickString(feature.properties?.height);
 729:     const levels = pickString(feature.properties?.["building:levels"]);
 730: 
 731:     const detailParts = [secondary, height ? `${height} m` : undefined, levels ? `${levels} levels` : undefined]
 732:       .filter(Boolean)
 733:       .join(" · ");
 734: 
 735:     activePopup?.remove();
 736:     activePopup = new Popup({
 737:       closeButton: false,

 739:       offset: 18
 740:     })

 742:       .setHTML(
 743:         `
 744:           <div class="popup-card">
 745:             <strong>${escapeHtml(title)}</strong>
 746:             ${detailParts ? `<span>${escapeHtml(detailParts)}</span>` : ""}
 747:           </div>
 748:         `
 749:       )
 750:       .addTo(mapInstance);
 751:   });
 752: 
 753:   mapInstance.on("error", () => {
 754:     statusPill.textContent = "Some external map tiles failed to load. The globe will keep trying.";
 755:   });
 756: }

 758: function wireSearch(): void {
 759:   searchForm.addEventListener("submit", async (event) => {
 760:     event.preventDefault();

 762:     if (query.length < 2) {
 763:       searchMessage.textContent = "Search needs at least two characters.";
 764:       searchResults.replaceChildren();
 765:       return;
 766:     }
 767: 
 768:     searchMessage.textContent = "Searching open place data…";
 769:     searchResults.replaceChildren();
 770: 
 771:     try {
 772:       const results = await searchPlaces(query);
 773:       if (results.length === 0) {
 774:         searchMessage.textContent = "No matching places came back from the public geocoder.";
 775:         return;
 776:       }
 777: 
 778:       searchMessage.textContent = `Found ${results.length} result${results.length === 1 ? "" : "s"}.`;
 779:       renderSearchResults(results);
 780:     } catch (error) {
 781:       searchMessage.textContent =
 782:         error instanceof Error ? error.message : "Search failed against the public geocoder.";
 783:     }
 784:   });
 785: }
