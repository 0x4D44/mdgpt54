## Missing Or Mismatched Items

**One item to verify — altitude source selection at `aircraft3dLayer.ts:5791`:**

The subagent found: `MercatorCoordinate.fromLngLat([track.lng, track.lat], track.altitudeMeters)`

The HLD requires (lines 379–381):
> use `geoAltitudeMeters` when present; otherwise `altitudeMeters` as visual fallback only

If line 5791 passes `track.altitudeMeters` unconditionally and the `geoAltitudeMeters`-first selection happens elsewhere before this call, the HLD is satisfied. But if `track.altitudeMeters` is the only altitude path at that call site without a prior `?? geoAltitudeMeters` pick, this is a real gap. The audit packet does not show the full altitude-selection expression at that line, so this cannot be confirmed from the packet alone. **Flag for manual verification.**

**Popup altitude-source disclosure:** The HLD says (line 384): "The popup should make the altitude source explicit when needed." The audit packet does not show the popup rendering clearly distinguishing geo vs. baro altitude in the output. The test coverage at lines 2580–2596 covers field order but not this annotation. Cannot confirm from packet alone. **Flag for manual check.**

No other gaps were found.

## Verdict

**Implementation is complete for the scoped HLD design.** All three steps are implemented: richer live metadata with 2D silhouettes (Step 1), static `icao24` identity lookup with 256 shards (Step 2), and bounded procedural 3D aircraft layer with hysteresis thresholds and lazy Three.js loading (Step 3). The single shared `LiveTrack` model is preserved, no new backend was added, and the GitHub Pages path is intact.

Two items could not be fully confirmed from the packet alone and should receive a quick manual check:
1. Whether `geoAltitudeMeters` is preferred over `altitudeMeters` in the actual `fromLngLat` call (`aircraft3dLayer.ts:5791`).
2. Whether the popup annotates the altitude source when barometric fallback is in use.

Everything else matches the HLD exactly.
