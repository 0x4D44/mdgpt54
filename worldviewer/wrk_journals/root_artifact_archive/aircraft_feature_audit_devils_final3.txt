## Summary

The audit packet covers all three steps: the HLD, six core implementation files, the generation script, and their test suites. The overall shape of the implementation is faithful Ś one shared `LiveTrack` pipeline, no second aircraft backend, 256 static JSON shards, bounded procedural 3D meshes, and lazy Three.js loading.

The analysis below flags only concrete plan-vs-implementation deviations and unstated assumptions that could cause real problems.

---

## Critical Gaps

**Gap 1 Ś Integration tests for `generateAircraftIdentity.ts` are absent from the packet**

HLD validation plan, line 440: *"unit tests for CSV-to-shard generation."*

The packet includes `aircraftIdentityData.test.ts`, which tests the CSV-line parser, header-index builder, and shard-tuple extractor. But there is no `generateAircraftIdentity.test.ts` visible anywhere in the packet. The following paths through the script remain untested by the evidence:

- `generateAircraftIdentity()` end-to-end (file I/O, full-shard write loop, prune logic)
- `formatShardSizeLine()` and the size-warning thresholds (`RAW_WARNING_BYTES = 5 MB`, `GZIP_WARNING_BYTES = 1.5 MB`)
- `pruneStaleShardFiles()` Ś this function **deletes files** from `public/aircraft-identity/`; an off-by-one in `REQUIRED_OUTPUT_DIR_NAME` guard could silently wipe the wrong directory

Whether this test file exists outside the packet is unknown. The absence is notable because the generation script is the only path that populates the shards the browser fetches.

---

**Gap 2 Ś 3D aircraft are click-silent; their invisible 2D counterparts respond to clicks instead**

`trafficLayers.ts:4456¢4472` registers `map.on("click", AIRCRAFT_LAYER, ...)` against the 2D GeoJSON source. Hidden aircraft (opacity=0) are still present as GeoJSON features and remain hittable. At zoom 10.5+ and pitch 45+, an aircraft at 10,000 m renders its 2D symbol far from its 3D mesh in screen space (the 2D symbol sits at the ground-projected lat/lng; the 3D mesh is visually elevated by altitude). A user clicking on the visible 3D mesh gets no response; a click near the ground position of the same aircraft triggers a 2D popup. The HLD is entirely silent on popup interaction with 3D aircraft.

This is not a code bug Ś it is a missing interaction that the design never addressed.

---

**Gap 3 Ś OpenSky poll errors clear all aircraft with no backoff**

`trafficClient.ts:3971¢3977`: any non-abort fetch error sets `this.latestAircraft = []` and `aircraftRuntime = "error"`, then the standard 15-second timer restarts. A 429 rate-limit or transient 503 produces a blank sky for 15 seconds and no user-visible explanation. No `Retry-After` parsing, no exponential backoff, no jitter. The HLD noted OpenSky reliability as a risk but specified no error-handling policy. This is a silent degradation path, not a plan contradiction.

## Assumptions To Challenge

**A1 Ś Anonymous OpenSky category data is assumed stable**

HLD line 144: *"manually verified as of 2026-03-12."* If OpenSky removes category for unauthenticated callers, every aircraft falls back to `aircraftVisualCategory = "generic"` and `Aircraft3dClassKey` defaults to `"narrow-body"`. Both steps 1 and 3 degrade silently. There is no telemetry, no warning to the user, and no code path that surfaces this change.

**A2 Ś 256 shards are assumed well-distributed without empirical validation**

HLD line 469 acknowledges shard skew. The generation script warns at thresholds but there is no evidence in the packet that a real OpenSky CSV has been run and the output sizes reviewed. Prefixes like `"38"¢"3f"` (French registry) and `"40"¢"4f"` (former CIS) are known to be dense. A first run of `refresh:aircraft-identity` could produce several shards far exceeding the warning threshold.

**A3 Ś `updatedAt` is set to fetch time, not OpenSky `last_contact`**

`openskyDirect.ts:2760`: `updatedAt: now`. The OpenSky state vector exposes `last_contact` at index 4 (the timestamp when OpenSky last heard from the transponder). At a 15-second poll interval, an aircraft that OpenSky last heard from 30 seconds ago looks fresh for a full poll cycle. The staleness/opacity fade (`STALE_THRESHOLD_MS = 60 s`) is meaningful only if `updatedAt` reflects actual signal age, not fetch time. HLD line 26: *"The design must stay honest about what the data really tells us."* This is a mild contradiction that was never called out.

**A4 Ś Step 3 go/no-go spike results are asserted, not demonstrated**

HLD lines 299¢309 enumerate three spike questions and assert all three answers were "yes." The implementation ships the full Step 3 on this basis. There is no spike test file, no benchmark output, and no recorded fps measurement in the packet. The go/no-go criteria ("keeps interactive frame rate comfortably interactive in representative high-pitch views") are expressed in prose only. If the procedural meshes or Three.js overhead are worse than expected on a low-end device, there is no automated path to detect this.

---

## Verdict

**No critical plan violations. The implementation is structurally sound and closely follows the HLD.**

All specified thresholds, field mappings, shard strategy, activation hysteresis, lazy loading, and fallback behaviours are present and tested. The shared `LiveTrack` pipeline was not forked. `SnapshotMessage`, `tracksToGeoJSON`, and popup rendering were not duplicated.

The two actionable gaps are:
1. Verify that `generateAircraftIdentity.test.ts` exists in the repository (not just the audit packet). If it is absent, the file-I/O paths in the generator are untested.
2. Decide whether clicking on a 3D aircraft should work, and if so, document the current limitation or wire the 3D layer to emit a click event.

Everything else is acceptable for a personal app at the stated scope.
