## Summary

All three steps are implemented. Steps 1 and 2 are faithful to the HLD in every measurable way. Step 3 is functionally present and correctly wired, but contains several real divergences from the HLD's stated process and specifications. None of the gaps in Steps 1ñ2 are critical. Three gaps in Step 3 warrant attention.

---

## Critical Gaps

### GAP 1: The Step 3 spike gate was bypassed

**HLD** (lines 63ñ68, 296ñ302, 436ñ437):
> "Step 3 should not start as a full feature build. Start with a narrow proof-of-concept that answers three questionsÖ If the answer to any of these is `no`, stop at Step 2."
> Spike deliverable: "one concrete class-model asset set identified."

**Implementation**: `aircraft3dLayer.ts` and `aircraft3d.ts` are a full production implementation ó `Aircraft3dRuntimeLayer`, all six class prototypes, `Aircraft3dController`, lazy loading, complete test suite. There is no documented spike result and the three HLD gate questions are never formally answered anywhere in the codebase. The implementation treats the procedural-geometry approach as an implicit positive answer, but this was not stated.

This is the largest plan-vs-implementation gap: a mandatory decision gate was bypassed.

---

### GAP 2: No glTF loader ó procedural meshes substituted without HLD approval

**HLD** (line 308):
> "Load three.js **and the glTF loader** dynamically the first time 3D aircraft are actually needed."

**Implementation** (`aircraft3dLayer.ts:45`):
```ts
constructor(Ö, loadThree: ThreeLoader = () => import("three"))
```
Three.js is loaded dynamically. No `GLTFLoader` is ever imported or loaded. All six class models are procedural Three.js geometry (`CylinderGeometry`, `BoxGeometry`, etc.) with no `.glb` asset files.

This is a valid architectural improvement ó it eliminates the asset-licensing gate entirely ó but the HLD explicitly required identifying "the specific model pack or files we will use" before proceeding past the spike. The implementation substitutes a different approach without stating why or obtaining that explicit sign-off. The HLD spike validation item "one concrete class-model asset set identified" is simply absent.

---

### GAP 3: No FPS performance acceptance test

**HLD** (lines 395ñ399):
> "Acceptance target: keep interactive frame rate above roughly 45 FPS in a representative high-pitch aircraft view on Arthur's normal browser machine. If that target is missed: lower the cap, raise the zoom threshold, reduce model complexity."

**Implementation**: There is no FPS measurement anywhere in `aircraft3dLayer.ts` or `aircraft3d.ts`. The hysteresis count thresholds are correctly implemented (24/32 aircraft), but the actual FPS acceptance criterion has no measurement mechanism and no evidence it was verified. The HLD frames this as a conditional acceptance test; the condition is unchecked.

---

### GAP 4: `onGround` field added to `LiveTrack` without HLD specification

**HLD** `LiveTrack` type (lines 82ñ109) lists nine aircraft-only optional fields. `onGround` is not among them.

**Implementation** (`trafficTypes.ts:17`):
```ts
onGround?: boolean | null;
```
`onGround` is a real field parsed from the OpenSky state vector (`STATE_ON_GROUND = 8`, `openskyDirect.ts:10`) and is semantically essential for Step 3 (suppresses grounded aircraft from `getAircraft3dAltitudeMeters`, `aircraft3d.ts:132`). Its addition is well-motivated, but it is an undocumented extension to the shared data model that the HLD explicitly enumerated.

---

### GAP 5 (minor): Altitude source is never labeled in the popup

**HLD** (line 379):
> "The popup should make the altitude source explicit when needed."

**Implementation** (`trafficLayers.ts:271ñ273`):
```ts
const altitude = formatAircraftAltitude({
  altitudeMeters: readNumber(props.altitudeMeters),
  geoAltitudeMeters: readNumber(props.geoAltitudeMeters)
});
```
`formatAircraftAltitude` silently prefers `geoAltitudeMeters` with no source label. The popup never emits "(geo)" or "(baro)" to distinguish which altitude is shown. This is a direct HLD non-compliance, though minor in practice.

## Assumptions To Challenge

**"The procedural-mesh approach implicitly satisfies the HLD's asset-identification requirement."**
Challenge: It does not. The HLD's spike gate was specifically about proving redistribution-safe assets exist *before* committing to Step 3. Replacing the entire asset model with procedural geometry is a different design decision that changes the go/no-go criteria entirely. The correct path was to decide this change *before* building, not after.

**"The count thresholds (24/32) are sufficient to guarantee acceptable frame rate."**
Challenge: Aircraft count is a proxy metric, not the actual acceptance criterion. A scene of 24 wide-body procedural meshes at high pitch may still hit the FPS target, or it may not ó on Arthur's specific browser machine, with the specific GPU, under real MapLibre rendering load. The HLD required empirical verification; no such measurement exists.

**"The geoAltitudeMeters coverage is good enough for Step 3 to be useful."**
Challenge: The HLD required sampling representative views to verify this before proceeding. The implementation correctly falls back to barometric altitude (`aircraft3d.ts:164`), but the HLD wanted quantitative evidence that the 3D view wouldn't be mostly empty. This was never sampled.

**"The HLD risks section saying '16 shards can still be uneven' is a document error."**
Confirmed ó the HLD's risks section (line 454) references 16 shards, but the design body specifies 256. This is a residue from an earlier design iteration in the HLD document, not an implementation gap.

---

## Verdict

Steps 1 and 2 are complete and correct. No critical gaps.

Step 3 is functionally implemented and structurally sound, but **three process requirements from the HLD were not followed**: the spike gate was skipped, the glTF-vs-procedural substitution was made without stated authorization, and the FPS acceptance criterion was never measured. These are real plan-vs-implementation gaps, not hypothetical risks.

The procedural-mesh approach is arguably better than glTF assets for this app ó no license hunting, no asset pipeline, smaller bundle, no external files to manage ó but that decision should be explicitly acknowledged as a deliberate deviation from the HLD, not a silent substitution. The FPS target should be spot-checked manually. The altitude-source popup label is a straightforward HLD item that was missed.
