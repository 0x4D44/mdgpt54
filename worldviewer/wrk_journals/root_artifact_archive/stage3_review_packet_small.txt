=== HLD Step 3 excerpt ===

Quarterly refresh is enough for this personal app. We do not need an automated ingestion pipeline.

### Stored Fields

Retain only:

- `icao24`
- `registration`
- `typeCode`
- `manufacturer`
- `model`

Drop everything else during generation.

### Static Artifact Strategy

Use 256 static JSON shards by first two hex characters:

```text
public/
  aircraft-identity/
    00.json
    01.json
    ...
    fe.json
    ff.json
```

Why 256:

- still mechanically simple and deterministic
- keeps the real OpenSky skew inside the shard-size budget in practice
- still static-host friendly for a personal app

### Generation Guardrails

The generation script must log output sizes for every shard before anything is committed.

If any shard is obviously too large for this app, stop and revise the shard scheme before shipping the refresh. A good warning threshold is:

- about `5 MB` uncompressed for a single shard, or
- about `1.5 MB` gzip for a single shard

We do not need automatic repartitioning. Manual review is enough.

### Runtime Behaviour

- when an aircraft becomes visible, look at the first two hex characters of `icao24`
- fetch that shard if it is not already in the session cache
- merge identity fields into matching visible aircraft
- if a fetch fails or an identity is missing, keep Step 1 behaviour

The cache is session-scoped only:

```ts
Map<string, Record<string, AircraftIdentity>>
```

No local storage. No persistence.

### Flight Number Handling

Step 2 still does not claim to know a true commercial flight number.

It only improves:

- registration
- type code
- manufacturer
- model

### Rendering Impact

This step unlocks better visual selection:

- `B738` -> 737-family model key
- `A20N` -> A320-family model key
- `B77W` -> 777-family model key
- fallback to category class when type code is missing

## Step 3: 3D Aircraft At Altitude

### Goal

Render aircraft as actual 3D objects at altitude when the camera is close enough for that to matter.

### Go / No-Go Spike First

Step 3 should not start as a full feature build.

Start with a narrow proof-of-concept that answers three questions:

1. can we render one aircraft model at live altitude inside a MapLibre custom layer without breaking map interaction?
2. do we have at least six redistribution-safe, low-poly class models that look good enough?
3. do representative views contain enough aircraft with usable altitude data to make the feature visible?

If the answer to any of these is `no`, stop at Step 2.

### Rendering Technology

Use a MapLibre custom 3D layer.

Load three.js and the glTF loader dynamically the first time 3D aircraft are actually needed. Do not ship them in the initial application bundle.

Until the lazy-loaded 3D path is ready, keep showing 2D aircraft symbols. Do not show an empty transition state.

### Why Not A Bigger 3D System

We do not need:

- a second renderer
- a persistent aircraft scene graph across the whole world
- a large asset pipeline
- dozens of type-specific models up front

We only need a bounded custom layer that renders a small set of visible aircraft.

### Model Strategy

Start with six class models only:

- `narrow-body`
- `wide-body`
- `regional-jet`
- `bizjet`
- `prop`
- `helicopter`

Do not make Step 3 depend on type-specific `.glb` assets existing.

If later we find license-compatible, low-poly models for common exact types, we can add overrides on top:

- `B738`
- `A320`
- `A21N`
- `B77W`
- `B788`

But class models are the required path. Exact overrides are optional polish.

### Asset Rule

Only use free, redistribution-safe aircraft models.

Before Step 3 moves past the spike, identify the specific model pack or files we will use. If no acceptable class-model pack exists, Step 3 is deferred.

### Activation Rules

Use hysteresis so the view does not flicker between 2D and 3D.

Activate 3D only when all are true:

- zoom is at least `10.5`
- pitch is at least `45`
- visible unclustered aircraft count is at most `24`

Drop back to 2D when any are true:

- zoom falls below `10`
- pitch falls below `35`
- visible unclustered aircraft count rises to `32` or more

The visible-count check must be based on the raw aircraft track array filtered by viewport bounds, not rendered map features. That avoids cluster interference.

### Altitude Policy

Step 3 should use:

- `geoAltitudeMeters` when present
- otherwise `altitudeMeters` as a visual fallback only

At the zoom and pitch where Step 3 activates, the visual error from barometric fallback is acceptable for a representative personal-app view.

The popup should make the altitude source explicit when needed.

The Step 3 spike should sample a few representative views and record roughly how often `geoAltitudeMeters` is null. If the 3D view would usually contain very few renderable aircraft, defer Step 3.

### Positioning

- convert the aircraft position with `MercatorCoordinate.fromLngLat(lngLat, chosenAltitude)`
- rotate by heading / true track
- keep on-ground aircraft on the 2D symbol path

### Performance Budget

Step 3 is only acceptable if it stays cheap enough to matter.

Acceptance target:

- keep interactive frame rate above roughly `45 FPS` in a representative high-pitch aircraft view on Arthur's normal browser machine

If that target is missed:

- lower the cap
- raise the zoom threshold
- reduce model complexity

Do not keep the feature on by default if it clearly harms navigation.

## UI Changes

Extend the aircraft popup to prefer this order:

1. best-effort `flightCode`
2. raw `callsign`
3. registration
4. manufacturer + model
5. type code
6. category label

Do not add a new global toggle for Step 1 or Step 2.

For Step 3, keep automatic activation by camera state in the first pass. If that proves confusing in practice, a single local `3D Aircraft` checkbox is an acceptable follow-up, but it is not part of the initial design.

## Validation Plan

### Step 1

- unit tests for OpenSky parsing with `extended=1`
- unit tests for callsign-to-flight-code formatting
- manual check that GitHub Pages still shows aircraft

### Step 2

- unit tests for CSV-to-shard generation
- unit tests for shard load and merge logic
- manual check for missing-identity fallback
- manual check that shard fetch failure is non-blocking
- manual review of shard-size logs before committing a refresh

### Step 3 Spike

- one live aircraft rendered at altitude in a custom layer
- one lazy-load proof that 3D code is not in the initial path
- one concrete class-model asset set identified
- one manual sample of a few representative views to estimate usable altitude coverage

### Step 3 Full Build

- unit tests for model key selection
- manual browser checks for altitude placement, heading rotation, and LOD cutoff
- manual check that 2D aircraft remain visible until the 3D path finishes loading

## Risks

- callsign is not the same as a true airline flight number
- OpenSky category data could change or disappear for anonymous callers later
- OpenSky aircraft metadata is crowdsourced and can be stale or incomplete
- even 16 shards can still be uneven by prefix
- 3D aircraft can get expensive fast if thresholds are too loose
- Step 3 is still contingent on finding acceptable free model assets

## Recommendation

Proceed exactly in order:

1. richer metadata and 2D silhouettes
2. static `icao24` identity lookup
3. Step 3 spike, then the 3D layer only if the spike proves it is worth shipping

That keeps the design honest, keeps GitHub Pages viable, and avoids turning a personal app into an aviation platform.
