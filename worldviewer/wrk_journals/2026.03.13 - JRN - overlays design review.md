# Overlays Design Review Journal

## 2026-03-13 11:08

- Arthur asked for a supervised agent-team review loop on the overlays HLD. Fair request. The original draft had the familiar problem of sounding cleaner than it really was once I checked it against the actual code.
- I used four review roles this time: architecture reviewer, implementation pragmatist, testability reviewer, and devil's advocate. That mix was better than just asking for "more review" because each pass looked for a different failure mode.
- Feeling: alert, mildly suspicious. Overlay features are exactly where small UI ideas turn into awkward lifecycle bugs if nobody forces the draft to respect the existing map behavior.

## 2026-03-13 11:27

- Round 1 found the important problems quickly. The weather plan depended on `map.style.sourceCaches`, which is private MapLibre internals and not acceptable for a design doc. The draft also ignored the app's current zoom/projection switch, which meant the `Night` overlay would have made mercator street views dingy for no real gain.
- The devil's advocate also called out the weakest slice correctly: TomTom road traffic. It was the only part asking for a new key, account setup, and conditional UI behavior. That looked out of proportion next to presets, night, and weather.
- V1 keeps the road-traffic slice alive but demoted. I also tightened the design around public APIs, zoom-aware visibility, and separate overlay credits.
- Feeling: better. The draft stopped pretending every requested idea deserved equal status.

## 2026-03-13 11:46

- Round 2 was mostly a tie-break session. The architecture reviewer wanted a more generic overlay abstraction. The pragmatist pushed back, and I agreed with the pushback. Two overlays do not justify a framework, and `main.ts` can tolerate two more buttons without turning into a catastrophe.
- The one place I did choose to cut scope was road traffic. After rereading the repo instructions and the current app setup, it was obvious that keeping a key-gated fourth slice inside this HLD would be architecture theater. It complicates setup, muddies the UI, and is not needed to solve the problem at hand.
- V2 therefore drops road traffic from the shipping plan and keeps only one small shared helper: layer anchors.
- Feeling: comfortable. This is the point where the document got smaller and more honest at the same time.

## 2026-03-13 12:01

- Final pass came back clean. No reviewer found another critical issue after V2. What remained were normal implementation risks: RainViewer availability, dateline validation for the night polygon, and third-party style-anchor drift.
- I promoted V2 to the final `2026.03.13 - HLD - overlays and presets.md` file and kept V1/V2 on disk so the review trail is visible.
- The most useful outcome from the whole loop is not a fancy new pattern. It is the decision to ship the three zero-key slices and stop there.
- Feeling: satisfied, with a hint of relief. This one could easily have become a grab-bag of overlays and optional knobs. It no longer is.

## 2026-03-13 12:18

- Arthur asked for one more confidence pass, which was the right call. I found a few things that were still under-specified enough to bite during implementation even though the overall design direction was sound.
- The missing pieces were: style-load coordination for default-on / rapidly toggled overlays, async fetch races on weather disable, RainViewer's linked attribution requirement, and an integration trap with the current `trafficUI` insertion logic if the new scene overlay credit reused `.credit-note`.
- None of that changed the overall architecture, but it did justify another design revision. I updated the final HLD and preserved that rereview as V3.
- Feeling: more comfortable now. Not because the design got larger, but because the remaining uncertainty is down to ordinary implementation work instead of hidden design holes.
