# 2026-03-11 Earth twin build journal

## 13:42

- Started from an empty workspace, which is cleaner than inheriting a half-broken app but also means every technical choice matters.
- Chose plain Vite + TypeScript + MapLibre GL JS to keep the runtime lean. React here would mostly be ceremony.
- Confirmed an open-data stack that can plausibly hit the brief in a regular browser: OpenFreeMap for vector streets and buildings, EOX imagery for the satellite look, MapLibre demo terrain for DEM, and Nominatim for geocoding.
- Feeling: focused, slightly annoyed that "fully realistic street-level digital twin" is a bigger ask than the phrase sounds. Open public data can get us to strong street-scale navigation, not universal photogrammetry.

## 13:48

- Core app is built and compiling. The scene uses globe projection, terrain, realistic satellite imagery, preset city flights, public geocoding, telemetry, and click popups for roads/buildings/POIs.
- Hit a real style-spec bug during validation: I was serializing `layout` / `paint` as `undefined`, and I had copied a `sky` idea that MapLibre rejected at the root level. Fixed both instead of hand-waving.
- Headless Chrome confirmed the app shell, map canvas, attributions, and live telemetry state. The actual WebGL globe still renders as a dark canvas in headless, which is a familiar GPU limitation rather than a style parse failure.
- Also fixed a genuine layout issue the smoke test exposed: the map canvas was initializing at 300px tall until I made the viewport sizing explicit and forced an early resize.
- Feeling: calmer now. The implementation is honest about the open-data ceiling, and the browser-side wiring looks solid.

## 14:00

- Arthur found the first real runtime pain point: deep street zoom could bog the browser down under a carpet of POI icons and labels.
- Confirmed the worst offender in the upstream style is `poi_r20`, which starts at zoom 17 and adds a lot of tiny symbol work exactly where the camera is already expensive.
- Added an adaptive performance mode in the app: it hides the densest symbol layers once zoom/pitch enter the stress zone, and I also lowered the hard zoom ceiling to stay just shy of the provider's ugliest regime.
- Extracted the throttle rule into a tiny module so it could be tested instead of buried in event code. Added Vitest tests for the threshold behavior and bumped the app version to `0.1.1`.
- I still could not get headless Chrome to replay Arthur's exact deep hash view reliably, so the validation here is compile/test based plus a source-level diagnosis, not a perfect automated visual repro.
- Feeling: mildly irritated by headless browser flakiness, but the diagnosis is straightforward and the mitigation is defensible.

## 14:03

- Arthur reported the interaction problem still felt mouse-related after the first drag, which means the last fix was too narrow.
- Removed `cooperativeGestures` first, but that alone was not enough. The next suspicious bits were self-inflicted: continuous view-sync work on every `move` event and URL hash updates on every camera change.
- Reworked the runtime so view syncing is animation-frame scheduled instead of raw-event spam, removed hash syncing entirely, and switched the map into `mercator` once zoomed in past globe territory. That should be materially less fragile for street-scale interaction.
- Feeling: more convinced this was event/render churn than literal mouse handling. If Arthur still sees a dead UI after this, I need a browser console trace or I need to instrument the map events directly.

## 14:19

- Arthur asked for terrain heights, which is reasonable because we were already drawing DEM-based terrain without exposing any actual elevation numbers.
- Confirmed the installed MapLibre build supports `queryTerrainElevation()`. Added a `Terrain` metric to the telemetry panel that reports the DEM height at the current map center.
- Adjusted the reported value to divide out terrain exaggeration so the number reads as approximate real elevation rather than the visually exaggerated mesh height.
- Also made the telemetry refresh immediately when terrain/building toggles change, not only after the next camera move.
- Feeling: this one was pleasantly straightforward. Useful feature, small surface area.

## 14:34

- Arthur clarified the real ask: make the landform itself read as relief in the image, not just in a number.
- Added a proper relief overlay stack: shared DEM terrain source, hillshade, and contour lines with contour labels. This is much closer to the "wiremesh over textured terrain" intent without throwing away the current renderer.
- Also added a `Relief` toggle so Arthur can turn the overlay off if it feels too busy, and bumped the app version to `0.1.2` because `package.json` changed again.
- The cost is real: bundle size went up modestly because `maplibre-contour` is now in the client. I think it is justified, but if performance regresses on weaker machines, relief should be the first optional layer to disable.
- Feeling: happier with this direction. It is visually honest and technically incremental instead of turning into a rewrite.

## 16:27

- Arthur called out the uncomfortable truth: even with contours and hillshade, the terrain still looked too flat.
- Swapped the DEM over to AWS Terrarium terrain tiles, raised the terrain source max zoom to 15, and added dynamic terrain exaggeration so regional hill country gets far more vertical emphasis than city views.
- Also retuned the visual stack so relief has room to breathe: darker hillshade shadows, warmer highlights, stronger contour visibility, and lower satellite opacity when relief is enabled.
- This is the right next move inside the current architecture. If the Pentlands still do not read strongly enough after this, the remaining lever is better local elevation data, not more styling tricks.
- Feeling: better. This is finally attacking the real bottleneck.

## 18:14

- Arthur pointed out faint vertical bands after the terrain upgrade. That was a useful catch because the picture looked plausibly like tile seams, not bad source data.
- The strongest clue came from the installed MapLibre runtime itself: it warns that reusing the same `raster-dem` source for both 3D terrain and hillshade reduces rendering quality. We were doing exactly that.
- Split the DEM wiring into separate source IDs for terrain mesh and relief shading while still sharing the same cached Terrarium tiles underneath, then softened the hillshade a bit at the zooms where seams were most visible.
- Also fixed the stale on-screen credit note, which was still claiming "MapLibre demo terrain" long after the source swap.
- Feeling: relieved. This is a much more honest fix than just blurring the symptom away with flatter lighting.

## 18:31

- Arthur sent another screenshot and that mattered: the surviving bands sit underneath the contours and read more like oblique imagery tile boundaries than hillshade seams.
- Rather than guess again, I moved satellite opacity control out of the static style expression and into a runtime helper keyed off both zoom and pitch. Relief mode now suppresses satellite imagery much harder in the exact regional, high-pitch views where those stripes are most obvious.
- This is a tradeoff, not magic. It should make the tile grid far less visible while keeping enough imagery for realism, but if the upstream imagery itself has inconsistent tone between tiles, the only absolute fix is a different imagery source.
- Feeling: more confident. The new screenshot finally made the layer ordering clue obvious.

## 19:05

- Arthur pointed out the more important mechanism: the landscape itself looked translucent. On review, that was plainly true because I had forced the map background transparent and had just made relief-mode imagery much more transparent as well.
- Reversed course. Relief mode now keeps the satellite skin fully opaque again, and the base map background is opaque too, so internal terrain skirts or lower render passes should not leak through the surface.
- This is a better fit for how terrain should read anyway. Relief should come from hillshade and contours stacked on top of solid ground, not from seeing through the ground.
- Feeling: mildly annoyed with myself because the previous opacity-based fix was elegant in theory and wrong in practice.
