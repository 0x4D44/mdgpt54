# Aircraft Design Review Journal

## 2026-03-12 11:20

- Arthur asked for an agent-team review loop over the aircraft identity / 3D model HLD. Good call. The first draft was useful, but it still had several places where it was too easy to read confidence into hand-wavy assumptions.
- I set up three Claude review roles: architecture reviewer, implementation pragmatist, and devil's advocate. The main value so far is that they disagreed in useful ways instead of repeating each other.
- Feeling: focused and slightly wary. Aviation data looks deceptively simple until the exact-field and exact-identity questions show up.

## 2026-03-12 11:34

- First round found real issues. The big ones were: unresolved shared-type strategy, a too-hand-wavy Step 2 metadata plan, an under-specified altitude rule for 3D, and missing bundle/LOD constraints for three.js.
- I also verified one disputed point instead of letting the review spiral into theory: anonymous browser calls to OpenSky with `extended=1` are currently returning 18-field state vectors with CORS. That means category can stay in Step 1, but the HLD now needs the exact field map spelled out.
- V1 changes are now aimed at simplification, not more architecture: one widened `LiveTrack`, 16 static metadata shards instead of 256, geometric altitude only for 3D, lazy-loaded three.js, and explicit hysteresis for switching between 2D and 3D.
- Feeling: better. The design is moving toward something implementable instead of just impressive-looking.

## 2026-03-12 12:02

- Second review round was calmer. No critical architecture problems remained, but the team still called out a few things worth tightening: the 3D threshold overlapped with clustering, Step 3 still assumed assets would somehow appear, and the geo-altitude-only rule may have been too pure to be useful.
- I agreed with the pragmatic pushback on one point and rejected it on another. I dropped the model-key stability idea because it was needless state for cosmetic gain. But I kept the overall Step 3 ambition by turning it into an explicit go/no-go spike rather than pretending it is already a guaranteed shipping slice.
- V2 now raises the 3D zoom threshold past clustering, keeps 2D visible while the 3D path lazy-loads, adds shard-size guardrails, and makes Step 3 contingent on proving assets, altitude coverage, and MapLibre integration are all good enough.
- Feeling: comfortable. This is now an honest design instead of a promise that every nice idea will definitely survive contact with reality.

## 2026-03-12 12:17

- Final review pass came back clean: architecture reviewer, pragmatist, and devil's advocate all reported no remaining critical issues in V2.
- I promoted V2 to the final HLD filename and kept the V1/V2 variants on disk so the review trail is visible. That felt like the right balance between process and clutter.
- The most useful outcome from this whole loop is that Step 3 is no longer pretending to be inevitable. It now has a proper proof-of-concept gate, which is exactly what the earlier draft was missing.
- Feeling: satisfied. The design is smaller, clearer, and much less likely to send us off building the wrong thing.
