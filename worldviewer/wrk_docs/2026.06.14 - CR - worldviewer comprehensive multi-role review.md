## 1. Executive summary

Worldviewer is a **well-engineered personal project punching above its weight** — clean module seams,
dependency injection at the boundaries, `strict` TypeScript throughout (all three tsconfigs typecheck
**clean**), consistent XSS-escaping discipline, field-validated external JSON, and a genuinely thorough
unit suite (**817 unit tests pass**). The headline worry of a "1084-line god-file" is overstated:
`main.ts` is long but *shallow* (mostly HTML template + wiring).

Two classes of problem dominate:

1. **A strategic identity crisis** (the devil's advocate's central thesis, verified in code): the project
   is *built* like a hosted service — a 2.7k-LOC WebSocket relay, AISStream provider, Docker, nginx — but
   *deployed* as a static GitHub Pages site where `isStaticTrafficHost()` (`trafficHelpers.ts:248`)
   **hard-disables the ship feature**. So the entire server tier is dead code on the only deployment that
   exists. This reframes the severity of every server finding (see §6).
2. **Concrete defects** worth fixing regardless: a genuine GPU-memory leak in the 3D aircraft layer, a
   permanently-red test gate, a cluster of relay reliability/security gaps (*if* the relay is ever hosted),
   and a scattering of geo-math edge-case bugs (antimeridian, equinox).

**Ground-truth validation (run during review):**

| Check | Result |
|-------|--------|
| `tsc --noEmit` (client / scripts / server) | ✅ all clean |
| `vitest run` (unit) | ✅ **817 tests pass** |
| `vitest run` (process) | ❌ **exits 1** — globs the 10 Playwright `e2e/*.spec.ts` files, which throw `test.describe() … not expected here` |
| `npm run check` (the documented "routine validation path") | ❌ **red**, because of the above |

## 3. Detailed findings by theme

### 3.1 The 3D aircraft material leak (H1) — *verified*
`applyAltitudeTint` (`aircraft3dLayer.ts:509`) clones each visible aircraft's `MeshStandardMaterial` once
(`mesh.material = mesh.material.clone()`, guarded by `MATERIALS_CLONED_USER_DATA`). But every teardown path
only removes the object from the scene:
- `syncAircraftObjects` stale loop (`:270-282`): `scene.remove(object)` → `objects.delete(id)` — no dispose.
- `dispose()` (`:227`): disposes only `prototypes`, not the live/removed clones.

Geometry is shared with the prototype (so it must *not* be disposed there), but the cloned **materials are
orphaned** — one per distinct aircraft that ever entered the 3D handoff zone and got tinted. Over a
multi-hour session with ICAO24 churn this is a steady WebGL program/material leak. **Fix:** in the stale
path and `dispose()`, traverse removed objects and `.dispose()` any material whose
`userData[MATERIALS_CLONED_USER_DATA]` is set (leave geometry alone).

### 3.2 Test gate is red (H2) — *verified, exit 1*
There is no `vitest.config.*` and no `test` block in `vite.config.ts`, so vitest's default glob picks up
`e2e/**/*.spec.ts`; all 10 throw `Playwright Test did not expect test.describe()`. The 817 real unit tests
pass, but the documented `npm run check` exits non-zero — training everyone to ignore the result. **Fix
(one-liner):** add to `vite.config.ts`:
```ts
test: { include: ["src/**/*.test.ts","server/**/*.test.ts","scripts/**/*.test.ts"], exclude: ["e2e/**","node_modules/**"] }
```

### 3.3 Server reliability/security cluster (C1, H3–H5, M6–M8) — *conditional, see §6*
The relay is competently structured in places — stale-socket identity guards (`if (shipSocket !==
nextShipSocket) return`), graceful shutdown, stale-track eviction, **the AIS key never leaves the server or
hits a log** (verified), and thorough subscribe-message validation. But it is missing the table-stakes
hardening for a *publicly reachable, unauthenticated, keyed* relay: global crash guards, send-time
backpressure, payload/connection/rate caps, an origin allow-list, exponential reconnect backoff, and a
tighter (and cosine-corrected) bbox cap. `/health` over-discloses internal state through nginx. These are
the difference between "a single bad event takes everyone down / anyone can drain my AIS quota" and a
robust service — *but only matter if the relay is actually deployed.*

### 3.4 Maintainability: known and largely planned (H6, M9, M10)
The dominant structural debt is the **overlay lifecycle duplication** (H6): 4 overlays re-implement the
same timer/revision/abort state machine; the subtle double-token cancellation logic should live in one
`createPollingOverlay(...)` factory. `main.ts`'s **triplicated toggle lists** (M9) are a classic drift bug
surface and sit in the only untested module. Notably, the team's own `notes/` already contains HLDs for
decomposition, test quality, type safety, and config housekeeping — **this debt is known and designed-for**;
the gap is execution, and a half-done decomposition (HLD targeted ~150 lines; `main.ts` is 1084, the
`appShell.ts`/`presetUI.ts` extractions never happened) is arguably worse than none.

### 3.5 Geo-math edge cases (M1, L3, L4, N2)
A recurring blind spot: **the antimeridian and the poles.** Solar terminator degenerates at the equinox
(M1, *verified*); ISS trail, search `fitBounds`, flight-route arcs, and the relay bbox validator all
mishandle 180°-crossing (L3/L4); longitude isn't wrapped for the coordinate readout (N2). Individually
minor and mostly cosmetic, but collectively they say "add a shared `wrapLongitude`/antimeridian-split
helper and a poles/dateline test fixture."

## 4. What is genuinely solid (verified, no action)

Worth recording so the team doesn't "fix" healthy code:
- **Security discipline:** every external/user string routed to the DOM goes through `escapeHtml`
  (aircraft/ship/earthquake/building popups, search results, bookmark cards); the rest are static
  literals. Hash parsing is **allow-listed** (finite-number + range-clamped + boolean-only), so no
  prototype pollution / injection / open redirect. No client-side secrets; **no `navigator.geolocation`**.
- **External JSON is field-validated** through `(value: unknown) => Typed | null` guards (OpenSky, ISS,
  earthquakes, flight route, aircraft identity, relay snapshot container).
- **Bounded growth in the client:** aircraft trails (ring buffer 60 + age prune + eviction), ISS trail,
  on-demand-only 256-shard identity fetch with in-flight dedup + negative cache, single-flight OpenSky
  poll with AbortController + backoff, RAF-coalesced view/3D sync, debounced subscribe/hash.
- **Test fidelity:** tests drive real logic against thin hand-written fakes (stateful mock map, mock
  WebSocket), not mock-theater — so the suite is worth *fixing* (H2), not rewriting.
- **Relay correctness primitives:** stale-socket guards, key confinement, graceful shutdown, JSON-enveloped
  logging that neutralizes log-forging.

## 5. Devil's advocate — the bigger picture

The strongest critique is not line-level; it is **direction**. Verified in code:

- **The relay is an appendage disconnected from its body.** ~2.7k server LOC + Docker + nginx exist to
  power *ships*, which `trafficClient.ts` refuses to connect to on `github.io` (the only deployment).
  Aircraft work browser-direct and prove the relay is *only* for ships.
- **The 2D→3D Three.js subsystem (~2k LOC + a heavy dep) auto-disables under load** — exactly when a busy
  sky would be most impressive. Maximum complexity for a payoff most users never trigger.
- **Tests are voluminous over trivia and silent over risk:** the riskiest file (`main.ts`) has zero unit
  tests; e2e "end-to-end" tests stub every API and serve **empty tile bodies**, so nothing exercises the
  real live-data/terrain experience. High coverage measures wiring-matches-test, not product-works.
- **The whole experience is a tenant on free community APIs** the README itself says to "swap for
  production" — a latent rot schedule that, with no live-path test, will fail silently.
- **Scope is a feature junk-drawer** (9 toggles + overlays + a 20-item brainstorm backlog) burying the one
  distinctive thing: cinematic orbit-to-street navigation.

**Reconciling the tension** (the other reviewers say "fix H1/server"; the devil says "delete the
features"): both are right at different altitudes. *If* the 3D layer and relay stay, H1 and the server
cluster are real and should be fixed. But the higher-order question — should they exist in a static
personal demo? — should be answered **first**, because fixing a leak in a subsystem you're about to delete
is wasted effort. The devil's three highest-leverage moves:
1. **Pick an identity and cut to it.** Declare it a static Earth-twin demo; move the relay/ships/Docker/nginx
   to an optional sidecar repo and drop the `aircraft3d*` subsystem (or gate it behind an explicit
   "experimental" flag). ~5k LOC and a heavy dependency gone; the deployed thing finally matches the built thing.
2. **Re-aim the tests** at `main.ts` orchestration + a small un-mocked smoke check; thin the
   exact-color/layer-order pinning tests that ossify cosmetics.
3. **Freeze scope**, finish only the risk-reducing half of the decomposition (extract template/presets,
   shrink + test the orchestrator).

I find this analysis sound. It does not contradict the line-level findings — it *prioritizes* them.

`dispose()` / bail the `move` handler when traffic off), M3 (skip unchanged-source `setData`), L2 (drop

**If the relay is ever hosted:** C1 (global crash guards + try/catch each `send`), H3 (backpressure on
`bufferedAmount`), H4 (`maxPayload` + connection/rate caps), H5 (origin allow-list), M6 (exponential
backoff+jitter), M7 (tighter, cosine-corrected bbox cap), M8 (trim/restrict `/health`), L8/L9 (non-root
container, pinned `tsx`/compiled JS, ping-pong reaper).

**Maintainability (endorse the team's existing HLDs):** H6 (overlay factory), M9 (single toggle table +
test), M10 (`ExpressionSpecification` over `as any`), shared `guards.ts`, finish/close the `main.ts`
decomposition HLD honestly.

**Cross-cutting:** add a `wrapLongitude`/antimeridian helper + a dateline/poles test fixture to retire
M1/L3/L4/N2 as a family; add element-shape validation to the `parseSnapshot`/bookmarks/search boundaries
