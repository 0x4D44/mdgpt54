# Feature Ideas & Improvements — Worldviewer

**Date:** 2026-03-16
**Context:** Brainstormed during comprehensive codebase review. Ideas are grounded in the current architecture (~9,300 LOC TypeScript, MapLibre GL JS, Three.js, WebSocket relay).

---

## Visual & Rendering Enhancements

### 1. Atmospheric Scattering Shader
**Description:** Add a procedural atmosphere glow around the globe at orbit zoom (z0–z4) using a custom MapLibre layer with a fragment shader that simulates Rayleigh scattering. The current space backdrop uses static CSS gradients — a real atmosphere would make the globe-to-surface transition feel dramatically more cinematic.
**Complexity:** Medium
**Considerations:** MapLibre's `CustomLayerInterface` supports WebGL shaders. The shader only needs to render at low zooms, so it can be disabled entirely above z5. Needs careful blending with the existing satellite raster layer. Reference: Shadertoy atmospheric scattering examples.

### 2. Cloud Layer Overlay
**Description:** Add a global cloud cover layer using NASA GIBS (Global Imagery Browse Services) MODIS/VIIRS imagery as a semi-transparent raster overlay on top of satellite imagery. Updated every 3 hours. This complements the existing weather radar (which only covers precipitation at z0–z7) with a global visual layer visible at orbit zoom.
**Complexity:** Small–Medium
**Considerations:** NASA GIBS provides free WMTS tiles. The overlay would follow the same enable/disable lifecycle as the existing solar terminator and weather radar overlays. Tile URL pattern: `https://gibs.earthdata.nasa.gov/wmts/...`. Opacity should fade below satellite imagery at higher zooms.

### 3. Starfield Background
**Description:** Replace the static CSS radial-gradient star dots with a proper star catalog rendered on a canvas behind the map. Use the Hipparcos bright star catalog (~9,000 stars) projected onto the celestial sphere, rotating with sidereal time. At orbit zoom, stars would be visible around the globe edges.
**Complexity:** Medium
**Considerations:** Rendering into a background `<canvas>` avoids MapLibre integration complexity. Star positions need equatorial-to-horizon conversion based on the current map center. Only visible at z < 3, so performance cost is minimal.

### 4. Aircraft Trail Lines
**Description:** Draw fading polyline trails behind aircraft showing their recent path (last 2–5 minutes of positions). This gives a sense of flight direction and recent movement, especially beautiful when viewed from an angled camera with 3D aircraft models visible.
**Complexity:** Medium
**Considerations:** Requires accumulating position history per aircraft ID in the `TrafficClient`. The trail could be a separate GeoJSON source with `LineString` features and a time-based opacity gradient. Memory bounded by capping at N positions per aircraft and pruning expired entries.

### 5. Altitude-Colored Aircraft
**Description:** Color aircraft icons and 3D models by altitude band (e.g., green for ground level, yellow for low altitude, blue for cruise, purple for high altitude). This creates an instant visual hierarchy that makes traffic patterns more readable at a glance.
**Complexity:** Small
**Considerations:** Already have `altitudeMeters` and `geoAltitudeMeters` in the GeoJSON properties. Use a MapLibre `"interpolate"` expression on `icon-color` keyed to altitude. For 3D models, apply a tinted material. The `aircraftVisualCategory` system could be extended to include altitude-based tinting.

### 6. Sunrise/Sunset Gradient Band
**Description:** Enhance the solar terminator overlay with a gradient twilight band (civil, nautical, astronomical twilight) instead of the current sharp day/night boundary. The terminator already computes the subsolar point — extending it to multiple concentric boundaries at solar depression angles of 6°, 12°, and 18° would create a beautiful gradient.
**Complexity:** Medium
**Considerations:** The existing `buildNightGeometry` function in `solarTerminator.ts` computes terminator boundary points. Generating additional boundaries at offset angles and rendering them as stacked fill layers with decreasing opacity would create the gradient effect. No new data sources needed.

## Data Layers & Real-Time Feeds

### 7. Flight Route Display
**Description:** When an aircraft is clicked, fetch and display its planned flight route from origin to destination as a great-circle arc on the map. Use the OpenSky route API (`/api/routes?callsign=...`) which returns origin/destination airport ICAO codes.
**Complexity:** Medium
**Considerations:** Not all callsigns have route data. The great-circle arc should be computed client-side using the Haversine formula and rendered as a dashed GeoJSON `LineString`. Airport coordinates could come from a small static lookup (top 5,000 airports, ~100KB gzipped). Integrates naturally with the existing aircraft popup.

### 8. Airport Overlay
**Description:** Add an optional layer showing major airports with runway outlines and ICAO/IATA labels. At high zoom, show individual runways; at low zoom, show airport dots with codes. This contextualizes the aircraft traffic and makes it easy to spot approach/departure patterns.
**Complexity:** Medium
**Considerations:** OpenStreetMap has runway and taxiway data available via Overpass API or pre-built tiles. Alternatively, a curated static GeoJSON of the top 500 airports with runway geometries (~500KB) would be simpler and more reliable. The layer should appear below traffic but above satellite imagery.

### 9. ISS Live Position
**Description:** Show the International Space Station's real-time position as a special icon on the globe at orbit zoom. The ISS travels at ~7.7 km/s and completes an orbit every 90 minutes, making it a compelling real-time visualization at the globe scale where regular traffic isn't visible.
**Complexity:** Small
**Considerations:** `http://api.open-notify.org/iss-now.json` provides a free, no-auth API returning current lat/lng. Poll every 5 seconds. Render as a unique symbol on a dedicated GeoJSON source. Draw the orbital track as a great-circle projection. Only visible at globe zoom (z < 5).

### 10. Ocean Current Visualization
**Description:** Add an animated particle flow overlay showing major ocean surface currents. Particles drift along current vectors, creating a wind-map-style animation. This brings the ocean surface to life, complementing the existing ship traffic layer.
**Complexity:** Large
**Considerations:** NOAA's OSCAR (Ocean Surface Current Analysis Real-time) provides global current velocity data as NetCDF. Needs a WebGL particle system similar to the MapLibre wind layer examples. GPU-based particle advection would be needed for smooth animation at reasonable particle counts (~50K). Could use a custom layer with instanced rendering.

### 11. Earthquake & Seismic Activity
**Description:** Show recent earthquakes from the USGS Earthquake API as pulsing circles on the map, sized by magnitude. Include a time slider to scrub through the last 7 days. This adds a dramatic geophysical layer that connects to the terrain and tectonic features already visible.
**Complexity:** Small–Medium
**Considerations:** USGS GeoJSON feed (`https://earthquake.usgs.gov/earthquakes/feed/v1.0/summary/`) provides real-time data, no auth required. Earthquakes would be a GeoJSON source with circle layers using data-driven radius and color based on magnitude. Pulsing animation via `requestAnimationFrame` updating circle opacity.

## User Interaction & Exploration

### 12. Measurement Tool
**Description:** Add a click-to-measure mode that draws a geodesic line between two points and displays the great-circle distance, bearing, and estimated flight time. Useful for understanding scale when exploring from orbit to street level.
**Complexity:** Small
**Considerations:** MapLibre supports `click` events for point collection. Geodesic distance via the Vincenty formula. Render the line as a GeoJSON `LineString` with distance labels. Add a toolbar button to enter/exit measurement mode. Minimal state management needed.

### 13. Bookmarks / My Places
**Description:** Let users save custom camera positions (lng, lat, zoom, pitch, bearing) with a name and optional note. Bookmarks persist in `localStorage` and appear alongside the preset grid. This makes the app feel personal and encourages repeat exploration.
**Complexity:** Small
**Considerations:** The preset system already has the right data structure (`Preset` type with camera parameters). Add a "Save View" button that captures current camera state. Render saved bookmarks in the same grid layout as presets. Max ~50 bookmarks before localStorage size becomes a concern.

### 14. Time-of-Day Scrubber
**Description:** Add a time slider that lets users set any time of day/year to see how the solar terminator, shadows, and lighting would look. Instead of the default "now", scrubbing to different times shows how daylight changes across the globe. Beautiful for understanding seasonal daylight patterns.
**Complexity:** Medium
**Considerations:** The solar terminator already accepts a `getNow` option. Override this with the slider's selected datetime. Also adjust the satellite imagery brightness or apply a tint shader to simulate golden hour / blue hour. The time scrubber could show a 24-hour strip with the current UTC and local times.

### 15. Deep-Link URL Sharing
**Description:** Encode the current camera position, enabled layers, and active overlays into a shareable URL hash (e.g., `#lat=55.95&lng=-3.19&z=14.8&p=68&b=-20&terrain=1&night=1`). Loading the URL restores the exact view. Simple but powerful for sharing interesting views.
**Complexity:** Small
**Considerations:** Read/write `window.location.hash` on camera move (debounced). Parse on page load to override defaults. Keep the hash format stable to avoid breaking shared links. Include only non-default values to keep URLs short.

### 16. Keyboard Navigation & Accessibility
**Description:** Add keyboard shortcuts for common actions: `T` for terrain toggle, `N` for night, `W` for weather, `A` for aircraft, `S` for ships, number keys 1–6 for presets, `Escape` to close popup, `/` to focus search. Include ARIA live regions for status changes.
**Complexity:** Small
**Considerations:** Wire a global `keydown` handler with a focus guard (ignore when typing in the search input). The existing toggle system is already well-structured for programmatic control. Add `aria-pressed` to toggle buttons and `aria-label` updates for status text.

## Performance & Offline

### 17. Service Worker for Tile Caching
**Description:** Add a service worker that caches satellite imagery, vector tiles, and aircraft identity shards for offline/low-bandwidth use. After a user explores an area, they can revisit it without network access. The globe at orbit zoom would always work offline after the first load.
**Complexity:** Medium
**Considerations:** Use a cache-first strategy for satellite tiles (they don't change) and network-first for vector tiles (they update monthly). Aircraft identity shards are perfect for long-term caching. Vite has service worker plugins. Cache size should be bounded (~500MB max with LRU eviction).

### 18. Progressive 3D Model Loading
**Description:** Load Three.js lazily only when the user first reaches the 3D aircraft handoff zoom/pitch. Currently, the `Aircraft3dController` already lazy-loads Three.js, but the bundle still includes it in the initial chunk analysis. Use Vite's `import()` with explicit chunk naming to ensure Three.js is a truly separate chunk that's only fetched on demand.
**Complexity:** Small
**Considerations:** The dynamic `import("three")` in `aircraft3dLayer.ts:46` already achieves this at runtime. Verify with `vite build --report` that Three.js is in a separate chunk. If not, add `/* @vite-ignore */` or configure manual chunks in `vite.config.ts`.

## Analytics & Insights

### 19. Flight Statistics Dashboard
**Description:** Show a compact dashboard of real-time statistics: total visible aircraft, aircraft by category breakdown (commercial, private, rotor, etc.), average altitude, busiest flight corridors in the viewport. Updates with each traffic snapshot.
**Complexity:** Medium
**Considerations:** All data is already available in the `SnapshotMessage`. Aggregate in a new `trafficStats.ts` module. Render below the metric grid in the control dock. The aircraft category distribution uses the existing `getAircraftVisualCategory` mapping. Corridor analysis could use simple spatial binning of aircraft positions.

### 20. Ship Density Heatmap
**Description:** When ship traffic is active, render a heatmap layer showing ship density by area. This reveals shipping lanes, port approaches, and busy waterways even when individual ship icons overlap. Toggle between individual ships and the heatmap view.
**Complexity:** Medium
**Considerations:** MapLibre supports `heatmap` layer type natively. Transform the ship GeoJSON source data into a heatmap layer with intensity based on point density. The `weight` property could incorporate speed (stationary ships at anchor have different significance than moving vessels). Toggle with a UI control.

## Developer Experience

### 21. E2E Smoke Test
**Description:** Add a Playwright test that loads the app, verifies the map renders, clicks a preset, enables aircraft, and checks that the status changes. This catches integration issues that unit tests miss (style loading, WebSocket proxy, tile fetching).
**Complexity:** Medium
**Considerations:** Playwright supports intercepting network requests for deterministic tests. Mock the OpenSky API and AISStream WebSocket. Assert on DOM state (status pill text, toggle classes). Run in CI but not on every commit — too slow for the main test loop.

### 22. Dev-Mode Traffic Simulator
**Description:** Add a `--simulate` flag to the relay server that generates fake aircraft and ship positions moving along realistic paths, without requiring real API keys or network access. Useful for frontend development, demo recordings, and testing traffic UI states.
**Complexity:** Medium
**Considerations:** Generate synthetic LiveTrack objects with randomized positions within a configurable bbox, realistic headings, and plausible speeds. Simulate gradual position updates on a timer. Include edge cases: grounded aircraft, stationary ships, aircraft with/without identity data. Wire into `createTrafficRelayApp` as an alternative data source.

## Infrastructure & Backend

### 26. Server-Side OpenSky Relay
**Description:** Move OpenSky aircraft polling from the browser into the relay server, mirroring the AISStream ship pattern. The relay polls once on behalf of all clients, unions their bboxes, and fans out state vectors via the existing snapshot protocol. Eliminates per-client CORS issues and consolidates API quota.
**Complexity:** Medium
**Considerations:** The snapshot protocol already has an `aircraft[]` field the relay sends empty — fill it. `TrafficRelayCore.reconcile()` currently hardcodes aircraft as `unavailable`; flip that. Authenticated OpenSky credentials server-side unlock 100 req/min vs 10 anonymous. The bbox union logic already exists for ships.

### 27. Relay Health & Metrics Endpoint
**Description:** Add an HTTP `/health` endpoint reporting connected clients, active bbox, ship track count, AISStream state, uptime, and memory. Minimal observability without adding a metrics dependency.
**Complexity:** Small
**Considerations:** Route on `req.url` in the existing `createServer` handler. All data accessible via `relay.getClientCount()`, `relay.getActiveShipBbox()`, `relay.getShipTrackCount()`, `process.memoryUsage()`. Return JSON. No auth needed.

### 28. Delta Compression for Snapshots
**Description:** Instead of full ship/aircraft lists every heartbeat, send only deltas (added/updated/removed) since the client's last acknowledged snapshot. Cuts WebSocket bandwidth ~75% for busy areas where most tracks are unchanged between 5-second beats.
**Complexity:** Medium
**Considerations:** Add a monotonic `seq` to each `SnapshotMessage`; client echoes last received `seq`. Relay keeps per-client "last sent" for diffing. Fall back to full snapshot on reconnect. Consider "significant change" thresholds (heading > 2°) to suppress trivial updates.

### 29. Docker Compose Deployment
**Description:** Package the relay server + Vite static build + nginx reverse proxy into a `docker compose up` deployment. Trivially deployable on any VPS with zero Node.js knowledge required.
**Complexity:** Small
**Considerations:** Multi-stage Docker build: stage 1 `npm ci && npm run build`, stage 2 slim Node image. nginx serves static frontend and proxies `/traffic` WebSocket. Env vars (`AISSTREAM_API_KEY`, `TRAFFIC_PORT`) already work for Docker secrets. Add `.dockerignore`.

### 30. Structured Relay Logging
**Description:** Replace `console.log`/`warn` calls with a structured JSON logger including timestamps, levels, client IDs, and correlation IDs. Makes logs parseable by aggregation tools.
**Complexity:** Small
**Considerations:** The relay already DI's `log: Pick<Console, "log" | "warn">`. Zero-dependency: wrap with `JSON.stringify({ ts, level, msg, ...ctx })`. Assign each WebSocket client a short ID for correlation. Pretty-print in dev, raw JSONL in production.

## Additional Visual Ideas (from brainstorm)

### 31. Cinematic Orbit Mode
**Description:** A "screensaver" mode that continuously flies between preset locations with smooth camera arcs, dwelling 15–20s at each before transitioning through orbit altitude to the next. Hands-free orbit-to-street experience.
**Complexity:** Medium
**Considerations:** Chain `flyTo` calls with `map.once("moveend")` callbacks. `flyTo` with `speed: 0.4` and high `curve` gives cinematic arcs automatically. Pause on user interaction; resume after idle timeout. Great for kiosk/demo.

### 32. Coordinate Readout
**Description:** Show cursor lat/lng and terrain elevation in a compact readout, updating on mousemove. Click to copy in configurable format (decimal degrees, DMS, UTM).
**Complexity:** Small
**Considerations:** `map.on("mousemove", e => e.lngLat)` + `map.queryTerrainElevation(e.lngLat)`. Render near status pill. Throttle to ~10 Hz. Long-press on touch devices.

### 33. Elevation Profile on Click-Drag
**Description:** Shift-drag a line across the map to render a compact elevation cross-section chart in the control dock using `map.queryTerrainElevation()` at sampled points.
**Complexity:** Medium
**Considerations:** Must normalize via `normalizeTerrainElevation()` from `reliefProfile.ts`. Render as canvas sparkline. Only works with terrain enabled.

### 34. Ship Icons & Wake Trails
**Description:** Replace the `>` text symbol with proper canvas-drawn ship silhouettes (matching the aircraft icon pattern) with tapered wake trails proportional to speed.
**Complexity:** Small–Medium
**Considerations:** Mirror `createAircraftIcon` approach. Wake as short LineString behind each ship. Heading cone in front. Use `icon-rotate` instead of `text-rotate`.

### 35. Minimap Globe Inset
**Description:** Small fixed globe in the corner showing the viewport's position on Earth. Provides geographic context at street zoom. Click to jump.
**Complexity:** Small–Medium
**Considerations:** Second MapLibre `Map` instance (200×200px), globe projection, low zoom, no terrain. Update viewport rectangle on main map `move` events. Hide when already at orbit zoom.

## Improvement Ideas (Non-Feature)

### 23. Extract main.ts Into Focused Modules
**Priority:** High (per code review)
**Description:** Split the ~800-line orchestrator into `mapStyle.ts`, `searchUI.ts`, `presetUI.ts`, `metricUI.ts`, and `mapLifecycle.ts`. Each module becomes independently testable.

### 24. Shared Overlay Lifecycle Base
**Description:** Extract the common revision-token + load-handler + timer pattern from `solarTerminator.ts` and `weatherRadar.ts` into a shared `overlayLifecycle.ts` helper. This would reduce duplication and make adding new overlays faster.

### 25. Unified Error Reporting
**Description:** Replace the scattered `console.warn` calls with a centralized `reportError(source, error)` function that can be wired to the status pill, browser console, and optionally an external error tracker. Currently, aircraft poll failures, identity shard failures, and ship relay errors all have their own logging patterns.
