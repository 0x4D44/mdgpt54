# Codebase Tech-Debt Sweep

Date: 2026-03-13
Repo: `C:\language\mdgpt54\worldviewer`

## Scope

This was a static multi-agent sweep for:

- tech debt
- dead or effectively dead code
- duplicate code and duplicated architecture
- things that are obviously wrong

The sweep covered `src`, `server`, `scripts`, repo-root hygiene, tests, and validation workflow.

It did not try to redesign the system. The goal was to surface the highest-value cleanup and correctness problems without inventing speculative architecture work.

## Team

- Architecture / maintainability reviewer
- Runtime / UI correctness reviewer
- Tests / tooling / repo-hygiene reviewer
- Devil's advocate reviewer to challenge weak claims and add missed issues

## Confidence Notes

High confidence:
- confirmed by more than one reviewer, or
- grounded in directly cited code paths with clear user or maintenance impact

Lower confidence:
- mainly structural smell without a concrete failure mode

## Executive Summary

The strongest problems are not cosmetic.

The repo currently carries duplicate traffic architecture, duplicated traffic-domain types/parsers, several real user-visible traffic/UI bugs, weak boundary-level coverage around the most failure-prone runtime code, and a badly cluttered repo root. The biggest cleanup win is to pick one aircraft-ingestion architecture and delete the other. The biggest user-facing win is to fix the traffic toggle/status/popup/search behavior before doing any refactor.

The devil's-advocate pass agreed with most of the important findings and mainly pushed back on weaker "file too big" style claims.

## High-Confidence Findings

### 1. Duplicate aircraft ingestion architecture

The browser client polls OpenSky directly in `C:\language\mdgpt54\worldviewer\src\traffic\trafficClient.ts:181`, while the ship relay subscribe path explicitly sends `aircraft: false` in `C:\language\mdgpt54\worldviewer\src\traffic\trafficClient.ts:376`.

At the same time, the server still carries aircraft polling and aircraft relay logic in:

- `C:\language\mdgpt54\worldviewer\server\trafficRelay.ts:25`
- `C:\language\mdgpt54\worldviewer\server\trafficRelay.ts:172`
- `C:\language\mdgpt54\worldviewer\server\trafficRelayCore.ts:159`

Impact:

- two architectures to maintain for one feature
- dead or effectively dead server-side aircraft path for the shipped browser client
- duplicated tests, duplicated parser logic, and drift risk

This is the single biggest tech-debt item in the repo.

### 2. Shared traffic-domain model and parser logic are duplicated and already drifting

The same transport concepts are defined twice:

- `C:\language\mdgpt54\worldviewer\src\traffic\trafficTypes.ts:1`
- `C:\language\mdgpt54\worldviewer\server\trafficModel.ts:1`

- browser: `C:\language\mdgpt54\worldviewer\src\traffic\openskyDirect.ts:16`

### 3. Ships toggle availability UX is misleading

The current behavior does not cleanly communicate "this feature is unavailable here."

Relevant paths:

- unavailable state only matters while ships are enabled: `C:\language\mdgpt54\worldviewer\src\traffic\trafficRuntime.ts:71`
- toggle UI is driven from that status: `C:\language\mdgpt54\worldviewer\src\traffic\trafficUI.ts:87`
- the main toggle handler auto-disables ships after error/unavailable transitions: `C:\language\mdgpt54\worldviewer\src\main.ts:1000`

Impact:

- on static hosts or unavailable relay backends, the user can click `Ships` and the control effectively flips back off
- the UI does not leave behind a durable, per-feature explanation
- the current behavior conflicts with README wording that implies a stable unavailable state

This is an obvious user-facing bug, not just debt.

### 5. Search requests race and can render stale results

The search submit path launches async lookups without cancellation or a request token:

- `C:\language\mdgpt54\worldviewer\src\main.ts:808`

This is a classic correctness issue in async UI code.

This matters less than the duplicate architecture and missing runtime tests, but it is a real logic quality concern.

## Weak Or Overstated Findings

These came up but should not drive the plan:

- "bbox helpers are duplicated" is weaker than "the transport model and OpenSky parser are duplicated"
- duplicated test doubles are real but low leverage
- file size alone is not a useful finding unless paired with a concrete failure mode

## Themes

### Theme 1: Delete duplicate architecture before abstracting

The repo has duplication at the architecture level, not just at helper level.

### Theme 2: Fix traffic UX/state mismatches before refactoring

Several user-visible issues are small, concrete, and worth fixing ahead of structural cleanup.

### Theme 3: Put coverage around stateful runtime boundaries

The helpers are tested far better than the lifecycles that actually break.

### Theme 4: Reduce repo clutter and establish one trustworthy validation path

The current root and command story both create noise.

## Suggested Priority Order

1. Fix ships toggle/status/popup/search/stale-marker bugs.
2. Clean up validation and repo hygiene enough that future work is visible and reviewable.
3. Decide the single aircraft-ingestion architecture and delete the other path.
4. Share traffic-domain types/parsers after the architecture choice.
5. Only then split large composition/presentation files where it clearly reduces blast radius.
