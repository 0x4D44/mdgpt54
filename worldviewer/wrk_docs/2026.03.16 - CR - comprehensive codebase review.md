## Executive Summary

## 1. Correctness & Bugs

### 1.1 Failing Test — Weather Radar Duplicate Enable (Severity: Medium)

**File:** `C:\language\mdgpt54\worldviewer\src\overlays\weatherRadar.test.ts:520-541`
**Test:** "does not duplicate fetches or layers on repeated enable calls for the same map"

The test expects `fetchImpl` to be called once after two rapid `enable()` calls on the same map, but it's called twice. The issue is in `createWeatherRadarOverlay`: the idempotency guard at line 295 (`if (enabled && currentMap === map && !reassertCurrentMap) return`) only fires after the first `enable()` sets `enabled = true`. But the first `enable()` call kicks off an async `refresh()`, and the second `enable()` call arrives before the first refresh completes — so `enabled` is already true, `currentMap === map`, and the guard correctly short-circuits. However, the first `enable()` call already started a timer via `startTimer()` which fires `refresh()` again.

Looking more closely: the test calls `enable` twice synchronously, then `await flushAsyncWork()`. On the first call, `enabled` becomes `true`, `currentMap` is set, and `refresh` is called (fetch #1). On the second call, the guard at line 295 should short-circuit because `enabled && currentMap === map && !reassertCurrentMap`. But the mock map's `getSource` returns truthy for the weather radar source after the first call adds it, so `needsReassertion` returns false, meaning `reassertCurrentMap` is false. The guard should therefore fire and return early.

**Hypothesis:** The issue is timing-related — the first `enable()` starts the timer, the second enable is indeed guarded, but `startTimer` fires an additional `refresh()` before `flushAsyncWork()` resolves. Actually, looking at the code again: the timer interval is `updateIntervalMs` which defaults to `WEATHER_RADAR_REFRESH_INTERVAL_MS` (300,000ms), so a timer-based duplicate seems unlikely in a test with fake timers.

**Recommendation:** Investigate and fix this test failure. It may indicate a real idempotency issue in the overlay's enable path, or the test may need adjustment for the current async flow.

## 2. Architecture & Design

### 2.1 Strengths

**Clean module boundaries.** The traffic system is well-partitioned:
- `trafficTypes.ts` — shared contract (both browser and server import this)
- `trafficHelpers.ts` — pure functions (formatting, GeoJSON conversion, parsing)
- `trafficRuntime.ts` — runtime state machines (pure logic, no side effects)
- `trafficClient.ts` — orchestrator with network I/O
- `trafficLayers.ts` — MapLibre layer management
- `trafficUI.ts` — DOM manipulation

**Overlay pattern.** Both `solarTerminator` and `weatherRadar` follow the same enable/disable lifecycle with revision tokens, load handlers, and timer management. This is consistent and testable.

**Dual transport architecture.** Aircraft (browser-direct via OpenSky) and ships (server-relay via AISStream) are cleanly separated. The `TrafficClient` orchestrates both transports and publishes a unified snapshot. This lets aircraft work on static GitHub Pages while ships require the relay.

### 2.2 Areas for Improvement

**main.ts is a monolith (~800 lines).** `C:\language\mdgpt54\worldviewer\src\main.ts` handles map initialization, style construction, UI wiring, search, presets, terrain toggling, overlay management, traffic lifecycle, and metric display. It has no tests and many responsibilities. This is the single largest risk in the codebase.

**Recommendation:** Extract concerns into focused modules:
- `mapStyle.ts` — style specification builder
- `searchUI.ts` — search form, results, and geocoding
- `presetUI.ts` — preset grid and fly-to
- `metricUI.ts` — camera metrics display
- `mapLifecycle.ts` — map instance creation, event wiring, toggle dispatch

**No shared overlay protocol.** The solar terminator and weather radar overlays share the same lifecycle pattern (revision tokens, load handlers, timer cleanup) but don't share any base abstraction. This is fine at two overlays but would become duplication at three or more.

## 3. Type Safety

### 3.1 Strengths

- **Strict mode enabled** across all three tsconfig projects with `noUnusedLocals`, `noUnusedParameters`, and `noFallthroughCasesInSwitch`.
- **Clean compilation** — zero type errors across client, server, and scripts.
- **Good type narrowing** — the codebase consistently checks `typeof` before accessing properties (e.g., `parseOpenSkyStates`, `parseLatestWeatherRadarFrame`).
- **Discriminated unions** used effectively (e.g., `TrafficConnectionStatus`, `TrafficLayerStatusCode`).

### 3.2 Minor Issues

**`any` usage in AISStream provider.**
`C:\language\mdgpt54\worldviewer\server\providers\aisstream.ts:46,78` — `parsePositionReport` and `parseShipStaticData` accept `any`. These are parsing external JSON; `unknown` with type narrowing would be safer.

**Type assertions in trafficLayers.ts.**
`C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts:181-183` — `setData` is accessed via `"setData" in source` check followed by a type assertion. This pattern repeats several times. A shared type guard would reduce duplication.

**Feature properties cast to LiveTrack.**
`C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts:245,254` — `feature.properties as LiveTrack` is unsafe because MapLibre serializes properties through JSON (stripping types). The popup code already handles this correctly by reading individual properties with `readString`/`readNumber`, so the cast is just cosmetic noise.

## 4. Error Handling & Resilience

### 4.1 Strengths

**Abort controllers throughout.** The aircraft polling path (`trafficClient.ts`), weather radar fetching (`weatherRadar.ts`), and search requests (`searchRequestController.ts`) all use `AbortController` to cancel stale requests.

**Revision tokens for overlay lifecycle.** Both overlays use incrementing revision tokens to reject stale callbacks after disable/re-enable cycles. This is a robust pattern that avoids race conditions.

**Exponential backoff reconnection.** The ship relay WebSocket reconnection in `trafficClient.ts:397-408` uses exponential backoff (2s base, 30s max). Good.

**Stale track expiry.** `trafficRelayCore.ts:79-83` expires ship tracks older than 5 minutes, preventing phantom ships from lingering.

### 4.2 Improvements Needed

**No retry limit on aircraft polling failures.** `trafficClient.ts:212-219` logs a warning and sets `aircraftRuntime = "error"` on fetch failure, but the poll timer continues indefinitely. After sustained OpenSky outages, this generates continuous error logs. Consider backing off after N consecutive failures.

**Ship feed reconnect doesn't clear stale tracks immediately.** When the AISStream feed disconnects and reconnects, old ship positions may still be visible until they expire (5 minutes). The reconnect handler doesn't flush stale state.

## 5. Security

### 5.1 Strengths

- **XSS prevention.** `escapeHtml.ts` is used consistently in popup HTML construction (`trafficLayers.ts:313-323`). All user-facing text (search results, aircraft labels) goes through this sanitizer.
- **No credential exposure.** The `AISSTREAM_API_KEY` is only used server-side (`trafficRelay.ts:17`). The browser never sees it.
- **CSP-compatible.** No inline scripts, `eval()`, or dynamic script injection.
- **Input validation.** `isValidSubscribeMessage()` in `trafficRelayCore.ts:58-77` validates bbox bounds, coordinate ranges, and layer types before processing client messages.

### 5.2 Minor Notes

**Nominatim search input is URL-encoded but not length-limited.** In `main.ts`, the search query is passed directly to the Nominatim API. While the server handles this safely, extremely long queries could be wasteful. A client-side length cap would be a minor improvement.

## 6. Performance

### 6.1 Current Optimizations

- **Performance mode** (`detailProfile.ts`) hides dense symbol layers at high zoom/pitch to keep the map responsive.
- **Aircraft clustering** at lower zooms (`trafficLayers.ts:57-60`) with `clusterRadius: 40` and `clusterMaxZoom: 10`.
- **3D mode hysteresis** (`aircraft3d.ts`) prevents flickering between 2D and 3D modes with separate on/off thresholds for zoom, pitch, and aircraft count.
- **Debounced subscribe** (`trafficClient.ts:72`) — 300ms debounce on viewport change before sending subscribe messages.
- **Aircraft identity shard loading** — only loads the shard(s) needed for visible aircraft, not all 256.
- **requestAnimationFrame batching** for 3D sync (`aircraft3dLayer.ts:85-88`).

### 6.2 Improvement Opportunities

**main.ts re-evaluates everything on every `moveend`.** The `moveend` handler in `main.ts` runs projection resolution, terrain exaggeration, satellite opacity, performance mode detection, relief toggling, contour updates, night overlay sync, and traffic subscribe — all synchronously on every camera move. Some of these (e.g., projection mode, terrain exaggeration) only change at specific zoom thresholds and could be gated by a "dirty" check.

**GeoJSON full replacement on every snapshot.** `trafficLayers.ts:172-198` calls `setData()` with a complete FeatureCollection on every traffic snapshot (every 5-15 seconds). For large aircraft counts, diffing or using MapLibre's `updateImage`/`setFeatureState` could reduce GC pressure.

**Three.js prototype cloning.** `aircraft3dLayer.ts:259` clones the prototype mesh for each visible 3D aircraft. If an aircraft class changes (rare), the old clone is removed from the scene but not explicitly disposed — `three.js Mesh.copy shares geometry/material references`, as the comment notes. This is correct but could accumulate `Object3D` wrappers.

**Weather radar tile source maxzoom is 7.** `weatherRadar.ts:14` caps the radar source at zoom 7, meaning radar tiles become very pixelated at higher zooms. This may be a RainViewer limitation but is worth checking if higher-resolution tiles are available.

## 7. Test Quality

### 7.1 Strengths

**214 tests across 22 test files.** Every non-main module has a corresponding test file. Coverage is thorough:
- Pure function tests (formatting, parsing, classification)
- State machine tests (runtime status resolution, connection status)
- Integration-style tests with realistic mocks (TrafficClient with MockWebSocket, overlay tests with MockMap)
- Lifecycle tests (enable/disable, reconnection, stale request rejection)
- Edge cases (empty inputs, null positions, malformed JSON, AIS sentinel values)

**Test isolation.** Each test file properly restores mocks/stubs in `afterEach`. The `FakeSocket`/`MockMap` patterns are consistent across files.

**Real WebSocket test.** `trafficRelay.test.ts:243-286` tests the full relay server lifecycle with a real `ws` WebSocket connection, including graceful shutdown. This is excellent.

### 7.2 Gaps

**main.ts has zero tests.** As the largest file (~800 lines), it's the only significant untestable module. This is a direct consequence of its monolithic design.

**aircraftIconSizing.ts has no dedicated test.** The icon sizing expression is covered transitively by the 3D handoff tests, but the zoom interpolation isn't directly tested.

**No visual regression tests.** Understandable for this project size, but the 2D/3D handoff and overlay stacking are visual behaviors that are hard to catch with unit tests alone.
