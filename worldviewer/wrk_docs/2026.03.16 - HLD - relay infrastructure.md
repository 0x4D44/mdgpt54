# HLD -- Relay Infrastructure

## Overview

Three small, independent improvements to the Worldviewer relay server: a health endpoint for observability, structured JSON logging for debuggability, and a Docker Compose setup for repeatable deployment. All three stay within the existing zero-dependency server philosophy -- no new npm packages, no feature flags, no migration choreography.

## Features

### 1. Relay Health Endpoint

#### What it does

Adds an HTTP `GET /health` route to the existing `createServer` handler in `trafficRelay.ts`. Returns a JSON object describing relay state so that operators (or Docker healthchecks) can inspect the process without connecting a WebSocket. Unauthenticated, always 200 -- the relay is healthy if it can respond.

#### Design

**Files to modify:**

- `server/trafficRelay.ts` -- route on `req.url` inside the `createServer` callback in `startTrafficRelayServer`. The current catch-all `200 text/plain` response becomes the fallback for unrecognized paths.
- `server/trafficRelay.test.ts` -- integration test hitting `/health` on a real HTTP server (same pattern as the existing `startTrafficRelayServer` shutdown test).

**Response shape:**

```
GET /health  ->  200 application/json

{
  "clients":       number,       // relay.getClientCount()
  "activeShipBbox": Bbox | null, // relay.getActiveShipBbox()
  "shipTracks":    number,       // relay.getShipTrackCount()
  "aisstream":     "disconnected" | "connecting" | "connected",
  "uptimeSeconds": number,       // Math.floor(process.uptime())
  "memoryMB": {
    "rss":       number,         // MB, rounded to 1 decimal
    "heapUsed":  number
  }
}
```

**Integration points:**

- `startTrafficRelayServer` already owns the `createServer` callback and has access to `app` and `options.relay`. The relay core exposes `getClientCount()`, `getActiveShipBbox()`, and `getShipTrackCount()` -- all synchronous, no new methods needed.
- AISStream connection state is not currently exposed. Add a `getAISStreamState()` accessor to `TrafficRelayApp` that returns `"disconnected" | "connecting" | "connected"` based on `shipSocket` and its `readyState`. This is the only new public surface.
- `process.uptime()` and `process.memoryUsage()` are standard Node APIs; no wrapper needed.

**Key function signatures:**

- `TrafficRelayApp` gains: `getAISStreamState(): "disconnected" | "connecting" | "connected"`
- No new types. The health response is a plain object serialized inline.

#### Test plan

- Unit test: call `getAISStreamState()` on a `createTrafficRelayApp` with a `FakeShipSocket` in each lifecycle phase (no socket, connecting, open, closed-and-reconnecting). Assert each returns the correct string.
- Integration test: `startTrafficRelayServer(0, ...)`, fetch `http://127.0.0.1:${port}/health`, assert 200, parse JSON, check that `clients` is 0 and `aisstream` is `"disconnected"`. Connect a WebSocket client, re-fetch, assert `clients` is 1.
- Negative: `GET /other` still returns the existing `200 text/plain` fallback (not a 404 -- keep the old behavior).

### 2. Structured Relay Logging

#### What it does

Replaces the relay's freeform `console.log`/`console.warn` calls with a thin structured JSON logger. Each log line is a single `JSON.stringify` call to stdout, making relay output machine-parseable for log aggregators without adding any dependencies. The logger slots into the existing `log: Pick<Console, "log" | "warn">` DI point, so all existing call sites and tests remain compatible.

#### Design

**Files to create:**

- `server/relayLogger.ts` -- the logger implementation.
- `server/relayLogger.test.ts` -- unit tests.

**Files to modify:**

- `server/trafficRelay.ts` -- import and use `createRelayLogger()` as the default value of `options.log` instead of bare `console`. Assign each WebSocket client a monotonic numeric ID for log correlation. Add the client ID to connect, disconnect, and subscribe log calls.

**Logger interface and factory:**

```ts
type RelayLogEntry = {
  ts: string;          // ISO 8601 timestamp
  level: "info" | "warn";
  msg: string;
  [key: string]: unknown;  // arbitrary context fields
};

type RelayLogger = Pick<Console, "log" | "warn"> & {
  info(msg: string, context?: Record<string, unknown>): void;
  warn(msg: string, context?: Record<string, unknown>): void;
};

function createRelayLogger(write?: (line: string) => void): RelayLogger;
```

The factory accepts an optional `write` sink (defaults to `process.stdout.write`) so tests can capture output without intercepting globals. The returned object satisfies `Pick<Console, "log" | "warn">` -- its `.log` method delegates to `.info`, its `.warn` method delegates to the warn-level path. This means it drops into the existing DI slot with zero call-site changes.

**Client ID assignment:**

In `trafficRelay.ts`, maintain a module-level `let nextClientId = 1` counter. When `handleClientConnection` is called, assign `const clientId = nextClientId++` and include `{ clientId }` in the connect, message, and disconnect log entries. The ID is short, monotonic, and scoped to the relay process lifetime -- no persistence needed.

**Log events (all as structured JSON lines):**

| Event | Level | Context fields |
|---|---|---|
| Client connected | info | `clientId`, `clients` (total count) |
| Client disconnected | info | `clientId`, `clients` |
| Client subscribe | info | `clientId`, `bbox`, `layers` |
| AISStream connecting | info | -- |
| AISStream connected | info | -- |
| AISStream disconnected | info | -- |
| AISStream resubscribed | info | `bbox` |
| AISStream error | warn | `error` (stringified) |
| Relay listening | info | `port` |

**Integration points:**

- `createTrafficRelayApp` already accepts `log` via `TrafficRelayAppOptions`. The default changes from `console` to `createRelayLogger()`.
- Existing tests pass `quietLog` which satisfies the same `Pick<Console, "log" | "warn">` interface. No test changes needed -- `quietLog` keeps tests silent.
- `startTrafficRelayServer` passes the same `log` through, so the structured logger covers both the app layer and the server bootstrap.

#### Test plan

- Unit test `createRelayLogger`: pass a `write` spy, call `logger.info("hello", { foo: 1 })`, assert the written string is valid JSON with `ts`, `level: "info"`, `msg: "hello"`, `foo: 1`.
- Unit test: `logger.log("x")` produces `level: "info"` (verifies the `Pick<Console, "log">` compatibility path).
- Unit test: `logger.warn("x", { err: "boom" })` produces `level: "warn"` with `err` in context.
- Unit test: `ts` field is a valid ISO 8601 string and `level` is one of the two allowed values.
- Existing `trafficRelay.test.ts` tests continue passing with `quietLog` -- no regressions.

### 3. Docker Compose Deployment

#### What it does

Packages the Worldviewer frontend and relay server into two containers behind nginx, so the whole stack runs with `docker compose up`. The frontend is a static Vite build served by nginx. The relay runs as a Node process. nginx reverse-proxies the `/traffic` WebSocket path to the relay container, providing a single entry point on port 80.

#### Design

**Files to create:**

- `Dockerfile` -- multi-stage build for both the static frontend and the relay server.
- `docker-compose.yml` -- two services (nginx + relay), one network.
- `nginx.conf` -- nginx config for static serving and WebSocket proxy.
- `.dockerignore` -- keep images small.

**Dockerfile (multi-stage):**

- **Stage 1 (`build`):** `node:22-slim`. Copy `package.json` and `package-lock.json`, run `npm ci`. Copy all source. Run `npm run build` (produces `dist/`). Run `npx tsc --project server/tsconfig.json` to compile server TypeScript to JS (or use `tsx` at runtime in the relay container -- see open questions).
- **Stage 2 (`frontend`):** `nginx:alpine`. Copy `dist/` from stage 1 into `/usr/share/nginx/html`. Copy `nginx.conf` into `/etc/nginx/conf.d/default.conf`.
- **Stage 3 (`relay`):** `node:22-slim`. Copy `package.json`, `package-lock.json`, run `npm ci --omit=dev`. Copy compiled server code from stage 1. Expose port 3210. `CMD ["node", "dist-server/trafficRelay.js"]` (or `CMD ["npx", "tsx", "server/trafficRelay.ts"]` if using tsx at runtime).

**docker-compose.yml structure:**

```yaml
services:
  relay:
    build:
      context: .
      target: relay
    environment:
      - AISSTREAM_API_KEY=${AISSTREAM_API_KEY:-}
      - TRAFFIC_PORT=3210
    expose:
      - "3210"
    restart: unless-stopped

  nginx:
    build:
      context: .
      target: frontend
    ports:
      - "8080:80"
    depends_on:
      - relay
    restart: unless-stopped
```

**nginx.conf:**

```nginx
server {
    listen 80;

    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
    }

    location /traffic {
        proxy_pass http://relay:3210/traffic;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400s;
    }

    location /health {
        proxy_pass http://relay:3210/health;
    }
}
```

The `proxy_read_timeout 86400s` keeps the WebSocket connection alive for long sessions. The `/health` proxy lets external monitors hit the relay health endpoint through the single nginx entry point.

**.dockerignore:**

```
node_modules/
dist/
.git/
wrk_docs/
wrk_journals/
notes/
*.txt
*.patch
*.md
```

**Integration points:**

- The relay container uses `AISSTREAM_API_KEY` and `TRAFFIC_PORT` environment variables, both already supported by `trafficRelay.ts`.
- The frontend Vite build uses `base: "./"` (already configured in `vite.config.ts`), so it works from any path root.
- The WebSocket client in the browser connects to `/traffic` relative to the page origin. In Docker, nginx proxies this to the relay container. In dev, Vite's proxy does the same thing. No client-side changes needed.

#### Test plan

- Manual: `docker compose build` succeeds without errors.
- Manual: `docker compose up`, open `http://localhost:8080` in a browser, verify the globe loads.
- Manual: `curl http://localhost:8080/health` returns the health JSON.
- Manual: enable ship layer in the UI with a valid `AISSTREAM_API_KEY` in `.env`, verify ship tracks appear.
- Manual: `docker compose down` cleans up cleanly.

## Implementation Order

1. **Structured relay logging** -- no dependencies, self-contained new file plus a small change to the default log in `trafficRelay.ts`. Write and test first because the health endpoint and Docker both benefit from structured logs being in place.
2. **Relay health endpoint** -- depends on nothing, but is more useful to verify once logging is structured. Small change to `startTrafficRelayServer`, plus one new accessor on the app object.
3. **Docker Compose deployment** -- depends on both features being merged so the container image includes them. Tested manually after the code features are in.

## Deployment Notes

- The Docker setup targets a single-machine dev/demo deployment. It is not designed for multi-replica or orchestrated environments.
- `AISSTREAM_API_KEY` should be passed via environment variable or a `.env` file in the project root. Docker Compose reads `.env` automatically. Do not commit the `.env` file.
- The nginx container exposes port 8080 by default. Change the `ports` mapping in `docker-compose.yml` to use a different host port.
- No volumes are needed -- the relay is stateless and the frontend is baked into the image.
- Rebuild after code changes: `docker compose build --no-cache && docker compose up -d`.

## Open Questions

1. **Server compilation strategy:** Should the relay Dockerfile compile TypeScript to JS via `tsc` (requires a server-specific `tsconfig` with `outDir`) or run TypeScript directly via `tsx` at runtime? `tsx` is simpler but adds ~15MB to the image and is a dev dependency. Leaning toward `tsx` for simplicity since this is a dev environment, but worth deciding before implementation.
2. **Health endpoint depth:** Should `/health` include the subscribed bbox per-client, or is the union bbox sufficient? The union bbox is simpler and avoids leaking per-client state. Leaning toward union-only.
3. **Log rotation:** Structured JSON to stdout means the container runtime handles log rotation (Docker's default JSON file driver). No application-level rotation needed, but worth confirming the Docker log driver config is acceptable for long-running dev deployments.
