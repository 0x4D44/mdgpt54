## Objective

Reduce the highest-value debt in the repo without inventing a new architecture.

The plan is intentionally biased toward:

- fixing user-visible correctness bugs first
- deleting duplicate code before abstracting it
- adding tests at failure-prone lifecycle boundaries before refactors
- improving repo hygiene and validation so future work is easier to trust

## Planning Principles

### 1. Delete before abstracting

The repo's biggest debt is duplicated aircraft architecture. Do not build a generic shared layer around two live implementations unless we first decide which one survives.

### 2. Fix concrete bugs before structural cleanup

Several traffic bugs are visible to users right now. Those are higher value than broad "file too big" refactors.

### 3. Put tests around runtime boundaries before changing them

`trafficClient.ts` and `trafficRelay.ts` are where async state and lifecycle bugs accumulate. Add enough boundary coverage that cleanup work has a safety rail.

### 4. Prefer one obvious validation path

A healthy repo should have one command Arthur can trust. The current split between client and server checks hides breakage.

### 5. Keep refactors scoped to the current problem

Do not turn this into a generic traffic framework, feature-flag exercise, or future-proofing project. Solve the debt that is here now.

## Phase 1: Correctness And UX Quick Wins

### Goal

Remove the obvious user-facing traffic and UI correctness bugs without changing overall architecture.

### Tasks

1. Fix ships unavailable-state behavior so the control can represent "enabled but unavailable" or otherwise communicate a durable per-feature failure state instead of silently flipping back.
2. Clear traffic popups when the corresponding layer is disabled, not only on full traffic clear.
3. Serialize or cancel search requests so stale results cannot overwrite newer input.
4. Clear or republish ship snapshot state when the relay closes so stale markers do not freeze on the map.
5. Replace the mojibake strings in `C:\language\mdgpt54\worldviewer\src\main.ts` with clean UI text.
6. Make aircraft-specific failure text explicit enough that users can distinguish aircraft feed failure from generic reconnecting.

### Acceptance Checks

1. Disabling `Ships` or `Aircraft` removes any popup tied to that layer immediately.
2. Reproducing a slower earlier search followed by a faster later search no longer shows stale results.
3. Disconnecting the ship relay clears or visibly invalidates ship markers.
4. The ships control presents a stable unavailable/error state rather than appearing to ignore user input.
5. UI strings render without broken encoding artifacts.

### Not In This Phase

1. No architecture changes to aircraft ingestion.
2. No `main.ts` decomposition unless required for one of the fixes above.
3. No generalized state-machine rewrite.

## Phase 4: Aircraft Architecture Simplification

### Goal

Delete the duplicate aircraft path and stop carrying two implementations for one feature.

### Decision Gate

Before coding, decide which aircraft ingestion path is the product path:

1. browser-direct OpenSky polling
2. relay-mediated aircraft delivery

The decision should be based on current product needs, deployment reality, credentials, and operational simplicity, not theoretical flexibility.

### Tasks

1. Choose the surviving aircraft path.
2. Delete the non-surviving path from production code, tests, and docs.
3. Remove any now-dead server or browser wiring.
4. After that deletion, create one shared traffic-domain module for the types and OpenSky parsing logic that still need to exist in both environments.
5. Reconcile README and developer docs with the chosen architecture.

### Acceptance Checks

1. Only one production aircraft ingestion path remains.
2. The duplicate traffic-domain model definitions are reduced to one shared source of truth where practical.
3. OpenSky parsing behavior is defined once, not maintained separately in browser and server copies.
4. The shipped browser path and the server path no longer disagree about which traffic features are active.

### Not In This Phase

1. No generic provider plugin system.
2. No migration or rollback support for the removed path.
3. No feature flags to preserve both implementations.

## Phase 5: Composition Cleanup Where It Pays Off

### Goal

Reduce blast radius only in the places where the earlier phases show repeated maintenance pain.

### Tasks

1. Split `C:\language\mdgpt54\worldviewer\src\main.ts` by feature wiring if the earlier fixes still leave search, traffic, overlays, or projection logic hard to reason about.
2. Split `trafficLayers.ts` or `aircraft3dLayer.ts` only if it directly improves testability or removes mixed responsibilities that caused bugs.
3. Avoid moving code solely to satisfy a style preference.

### Acceptance Checks

1. The resulting seams map to concrete responsibilities, not arbitrary file size targets.
2. The split reduces the amount of unrelated code touched by routine feature changes.
3. Tests become easier to write or more focused as a result.

### Not In This Phase

1. No broad app-shell rewrite.
2. No extraction of abstractions that exist only for symmetry.

## Devil's-Advocate Guardrails

During execution, keep challenging the work with these questions:

1. Are we fixing a real bug, or just rearranging code?
2. Did we delete anything meaningful, or only add layers around it?
3. Are we creating a framework because two files look similar, even though one of them should simply be removed?
4. Are we writing tests for behavior that matters, or tests that merely mirror implementation?
5. Are we about to start a refactor whose only justification is that a file feels large?

## Definition Of Done

1. the user-visible traffic/search issues are fixed
2. repo validation and hygiene are materially better
3. the risky traffic runtime boundaries have focused tests
4. only one aircraft ingestion architecture remains
5. any follow-on file splits are justified by reduced blast radius, not aesthetics
