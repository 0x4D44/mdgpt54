# Overlays and Presets

## Context

Worldviewer already provides:

- a launch-view preset grid in `src/main.ts`
- inline `Scene` toggles for terrain, relief, buildings, and spin
- a separate dynamic `Live Traffic` section for the more complex traffic controls
- automatic projection switching from globe to mercator at zoom `6` and back at zoom `5`

The reviewed 2026-03-12 draft had three critical problems:

1. the weather refresh path relied on private MapLibre internals
2. the overlay behavior ignored the app's zoom/projection model
3. the road-traffic slice was the only part that needed a new key-gated branch and added more setup than value

This reviewed HLD resolves those issues by tightening the design and cutting the weakest slice.

## What Matters

- Small code with obvious ownership.
- No backend work.
- No new dependencies.
- No feature flags.
- Minimal environment-variable use.
- Each shipped overlay should be useful in the scale range where the app actually operates.

## Non-Goals

- Weather animation
- A generic overlay plugin system
- Exact astronomical precision
- Commercial-grade uptime or coverage guarantees
- Road traffic in this implementation pass

## Decision Summary

Ship three additions:

1. curated presets
2. solar terminator
3. weather radar

Defer road traffic to a separate follow-up if Arthur later decides the extra key/account friction is worth it.

Cross-cutting decisions:

- Keep overlay logic in `src/overlays/*`.
- Keep `main.ts` changes small: add two buttons, one scene note, one overlay-credit line, and wire module calls.
- Add one tiny layer-anchor helper instead of a generic overlay framework.
- Respect the current zoom/projection model instead of forcing every overlay to render at every scale.

## Cross-Cutting Design

### Overlay lifecycle

Each overlay module owns:

- its source id
- its layer id
- any interval / fetch handles
- idempotent `enable(map)` and `disable(map)` functions
- a small async-safety guard such as an `AbortController` or revision token

Do not build a shared controller abstraction yet. Two focused modules do not justify it.

If `enable(map)` is called before the style is ready, the module should wait for the first `load` event and then apply once. `disable(map)` must clear timers, abort in-flight work, and prevent late async completions from re-adding layers after the toggle is off.

### Layer anchors

Add `src/overlays/overlayAnchors.ts` with small helper functions to find:

- the first label layer
- the satellite-imagery layer
- the first road layer

Scan the style in order and return the first matching layer. Known ids are recognition hints for styles that do not expose obvious text/road/raster signals, but they must not outrank an earlier matching layer. The goal is to keep layer placement deterministic without binding the whole feature to exact third-party style ids.

### Scene notes and overlay credits

Add two hidden elements below the `Scene` toggle grid:

- `scene-overlay-note`
- `scene-overlay-credit`

Do not rewrite the existing base credit paragraph. Keep overlay attribution separate, the same way traffic attribution is already separate.

Do not give the scene overlay credit the generic `.credit-note` class. The current traffic UI inserts itself before the first `.credit-note`, so reusing that class here would silently change section ordering. If shared styling is wanted, use a different utility class or give the base credit paragraph a stable id.

### Zoom-aware visibility

The app is not one uniform view. Its overlays should follow that.

- `Night` is globe-oriented and fades out completely by zoom `6`
- `Weather` is useful up to metro scale and stops rendering above zoom `8`

That removes the two most obvious UI failures from the original draft:

- a dark street-level mercator map
- blurry overscaled radar at street zoom

## 2. Solar Terminator

### Goal

Show the current night hemisphere as a soft orbit-scale overlay.

### Design

Add `src/overlays/solarTerminator.ts`.

The module:

1. computes the subsolar point from UTC time using simplified NOAA-style formulae
2. builds the night-hemisphere polygon around the subsolar antipode
3. adds a GeoJSON source plus `fill` layer
4. updates once per minute while enabled

### Rendering rules

- normalize longitudes to `[-180, 180]`
- insert the layer below the first label layer
- use zoom-based opacity so the mask fades out before the globe-to-mercator switch

Opacity target:

- `0.18` at zoom `0`
- `0.12` at zoom `4.5`
- `0` at zoom `6`

The original draft claimed that globe projection would make antimeridian issues disappear. That is too optimistic. Instead, the implementation should treat dateline behavior as something to test and verify.

### UI

- Add a `Night` chip to `Scene`
- Default: on

### Validation

- unit tests for declination sanity at known dates with tolerances
- unit tests for polygon validity and longitude normalization
- seam-aware unit tests that compare night/day classification against solar illumination on both sides of the antimeridian and across the March declination sign flip
- manual dateline check before shipping

### Files

- `src/overlays/solarTerminator.ts`
- `src/overlays/solarTerminator.test.ts`
- `src/overlays/overlayAnchors.ts`
- `src/main.ts`

## 3. Weather Radar

### Goal

Show the latest available RainViewer precipitation frame as an opt-in overlay.

### Source

Use RainViewer's Weather Maps API.

Verified review notes:

- the API is still available for personal / educational use
- only past radar frames remain after the 2025-2026 API transition
- maximum tile zoom is `7`
- RainViewer asks for source attribution with a link
- the frame `time` is frame-generation time, not an exact data timestamp

### Design

Add `src/overlays/weatherRadar.ts`.

The module:

1. fetches `https://api.rainviewer.com/public/weather-maps.json`
2. selects the newest `radar.past` frame
3. builds the tile url from `{host}` + `{path}`
4. adds a raster source and layer when enabled
5. polls metadata every 5 minutes only while enabled

When a newer frame appears, update the existing source with the public MapLibre API:

- `source.setTiles([nextUrl])`

Do not touch private style caches.

### Rendering rules

- source id: `weather-radar`
- layer id: `weather-radar`
- `tileSize: 512`
- source attribution should include an HTML link to `https://www.rainviewer.com/`
- opacity `0.6`
- insert below the first road layer; in the current style that keeps radar above satellite, hillshade, and contours while leaving roads and labels readable
- layer `maxzoom: 8`

The extra layer max zoom is deliberate. RainViewer stops at source zoom `7`; letting the map stretch that indefinitely would look broken.

### UI

- Add a `Weather` chip to `Scene`
- Default: off
- Show a status line such as `Radar frame 12:10 UTC - RainViewer` while active

Use a fixed UTC timestamp string. Do not add another timer just to keep "x minutes ago" fresh.

### Failure behavior

- if metadata fetch fails, hide the layer and show `Radar unavailable`
- if `host`, `path`, or `radar.past` is missing, treat that the same as unavailable rather than throwing
- retry on the next 5-minute cycle while the toggle stays on
- let tile failures degrade naturally to blank patches

### Attribution

When active, the scene overlay credit line should include `RainViewer`.

The required outbound link should live in the raster source attribution so MapLibre's attribution control carries the linked credit even if the scene note is plain text.

### Validation

- unit tests for metadata parsing and tile-url construction
- unit tests for source-update behavior around `setTiles`
- unit tests for empty / malformed metadata and for disable-while-fetching races
- manual check that repeated enable / disable cycles do not duplicate sources or layers

### Files

- `src/overlays/weatherRadar.ts`
- `src/overlays/weatherRadar.test.ts`
- `src/overlays/overlayAnchors.ts`
- `src/main.ts`

## Deferred: Road Traffic

Road traffic is no longer part of this HLD.

Reason:

- it is the only slice that needs a new key and account setup
- it is the only slice that creates a key-dependent UI branch
- it adds the most operational friction while solving the least urgent problem

If Arthur wants it later, write a separate small HLD for it. Do not let it complicate the simpler overlays now.

## Implementation Order

1. Presets
2. Solar terminator
3. Weather radar

## Validation Plan

### Unit tests

- solar math sanity checks with tolerances
- solar polygon shape / normalization
- RainViewer metadata parsing
- tile-url construction
- weather source update logic

### Manual checks

- `Night` is clearly visible at orbit scale and gone by zoom `6`
- `Weather` is useful at regional / metro scale and gone above zoom `8`
- active overlay credits appear only when needed
- dateline-adjacent globe views do not obviously tear the night polygon

## Risks

- RainViewer has no SLA and explicitly does not guarantee permanent availability
- the night polygon can still show seam artifacts if the ring math is wrong, so antimeridian behavior needs explicit regression coverage instead of exact-boundary probe assumptions
- anchor helpers still depend on third-party style structure, so fallbacks matter

## Recommendation

Ship the three zero-key slices. Defer road traffic unless the added setup cost becomes justified by an actual use case.
