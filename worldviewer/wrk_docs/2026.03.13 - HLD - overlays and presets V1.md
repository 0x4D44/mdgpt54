# Overlays and Presets

## Context

Worldviewer already has:

- a curated launch-view preset grid in `src/main.ts`
- a `Scene` section with inline toggle chips
- a dynamic `Live Traffic` section with its own UI helper
- automatic globe-to-mercator switching at zoom `6` and back to globe at zoom `5`

Arthur asked for a careful review of the 2026-03-12 overlays draft. The first review pass found real issues in the original version:

- the weather refresh plan depended on private MapLibre internals
- the draft ignored the app's existing projection switch, so the night overlay would darken street-scale mercator views
- layer insertion points were described loosely and would be brittle against style changes
- the road-traffic slice was the only one that added a new key-gated branch of behavior

This `V1` keeps all four requested additions in scope, but tightens the design so it fits the current app and public APIs.

## What Matters

- Visible value with small, boring code.
- No new backend.
- No new npm dependencies.
- No feature flags.
- Minimal environment-variable use.
- The map should stay pleasant from orbit down to street zoom.

## Non-Goals

- Weather animation timelines
- A weather product suite
- Precise sunrise/sunset calculations for legal or scientific use
- Commercial traffic-data reliability
- A reusable overlay framework

## Decision Summary

Implement four slices in this order:

1. curated presets
2. solar terminator
3. weather radar
4. optional road traffic

Cross-cutting decisions:

- Keep `main.ts` responsible only for HTML changes, state wiring, and toggle clicks.
- Put overlay behavior in `src/overlays/*`.
- Add one tiny helper for stable layer insertion points instead of hard-coding assumptions in each overlay module.
- Add one dynamic scene note and one dynamic overlay-credit line instead of mutating the existing base credit paragraph.

## Cross-Cutting Design

### Overlay lifecycle

Each overlay module owns:

- its source id
- its layer id
- any polling / interval handles
- idempotent `enable(map)` and `disable(map)` functions

No generic registry is needed yet. Two focused modules plus one optional road-traffic module are still small enough to keep direct.

### Layer anchors

Add `src/overlays/overlayAnchors.ts` with a few public helper functions:

- `findFirstLabelLayerId(style)`
- `findRoadAnchorLayerId(style)`
- `findSatelliteAnchorLayerId(style)`

Each helper should use the current OpenFreeMap style ids first, then fall back to a simple scan by layer type. The goal is to keep insertion deterministic without pretending third-party style ids are permanent.

### Scene notes and overlay credits

Do not rewrite the existing static bottom credit note.

Add two small elements under the `Scene` section:

- a hidden status line for overlay-specific notes
- a hidden credit line for active scene overlays

Examples:

- `Radar frame 12:10 UTC - RainViewer`
- `Overlays: RainViewer, TomTom`

This follows the same separation the current app already uses for traffic credits.

### Zoom-aware visibility

The original draft treated all overlays as equally valid at every zoom. That does not fit this app.

Use these visibility rules:

- `Night` fades out to `0` opacity by zoom `6`
- `Weather` renders only through zoom `8`
- `Road Traffic` renders from zoom `6` upward

This keeps each overlay in the scale range where it is actually useful.

## 2. Solar Terminator

### Goal

Show the current night hemisphere as a soft globe-scale mask.

### Design

Add `src/overlays/solarTerminator.ts`.

The module:

1. computes the subsolar point from the current UTC time using simplified NOAA-style formulae
2. generates the night-hemisphere polygon around the subsolar antipode
3. adds a `fill` source/layer pair
4. updates once per minute while enabled

### Important corrections from review

- The overlay should not stay visually heavy at street zoom.
- The design should not claim antimeridian handling is "free".

So:

- normalize output longitudes to `[-180, 180]`
- test a few dateline-adjacent timestamps
- use a zoom-based opacity expression so the layer fades out before mercator takes over

Opacity target:

- `0.18` at zoom `0`
- `0.12` at zoom `4.5`
- `0` at zoom `6`

### UI

- Add a `Night` toggle chip to the existing `Scene` section
- Default: on

### Files

- `src/overlays/solarTerminator.ts`
- `src/overlays/solarTerminator.test.ts`
- `src/overlays/overlayAnchors.ts`
- `src/main.ts`

## 3. Weather Radar

### Goal

Show the latest available precipitation frame as an opt-in overlay.

### Source

Use RainViewer's Weather Maps API.

Relevant facts verified during review:

- `weather-maps.json` still exists
- only past radar frames remain
- maximum tile zoom is `7`
- the `time` field is frame-generation time, not exact precipitation-observation time

### Design

Add `src/overlays/weatherRadar.ts`.

The module:

1. fetches `https://api.rainviewer.com/public/weather-maps.json`
2. selects the latest `radar.past` frame
3. builds a raster tile url from `{host}` + `{path}`
4. adds a `RasterTileSource` plus raster layer
5. refreshes metadata every 5 minutes while enabled

When a new frame arrives, update the source through the public MapLibre API:

- `source.setTiles([nextUrl])`

Do not touch `map.style.sourceCaches` or any other private internals.

### Presentation

- Source id: `weather-radar`
- Layer id: `weather-radar`
- `tileSize: 512`
- `maxzoom: 8` on the layer so the map stops stretching blurry tiles at street scale
- opacity `0.6`
- insert above satellite imagery and below vector roads / labels

### UI

- Add a `Weather` toggle chip to `Scene`
- Default: off
- When enabled, show a status line such as `Radar frame 12:10 UTC - RainViewer`

Use a fixed UTC timestamp string. Do not add a second timer just to keep a relative "3 min ago" label fresh.

### Failure behavior

- if metadata fetch fails, keep the toggle active but hide the layer and show `Radar unavailable`
- retry on the next 5-minute cycle
- if tiles fail, let MapLibre show gaps; do not add custom retry code

### Attribution

When the layer is active, the scene overlay credit line includes `RainViewer`.

### Files

- `src/overlays/weatherRadar.ts`
- `src/overlays/weatherRadar.test.ts`
- `src/overlays/overlayAnchors.ts`
- `src/main.ts`

## 4. Road Traffic

### Goal

Show live road-flow tiles at road zooms.

### Source

TomTom raster flow tiles:

- style `relative0`
- tile endpoint with `key={Your_API_Key}`
- recommended style for congestion-relative coloring

### Design

Add `src/overlays/roadTraffic.ts`.

The module:

1. reads `VITE_TOMTOM_API_KEY`
2. exposes `available`
3. adds a raster source/layer when enabled
4. removes them when disabled

No polling loop is needed because the tiles are already live.

### Key-handling decision

The original draft hid the toggle entirely when the key was missing. That creates an invisible product branch and makes the UI harder to reason about.

`V1` changes that behavior:

- always render the `Road Traffic` toggle
- if the key is missing, render it disabled with the label `Road Traffic`
- show a plain note like `Requires a local TomTom key`

That keeps the app honest while still limiting environment variables to one optional key.

### Presentation

- Source id: `road-traffic`
- Layer id: `road-traffic`
- `tileSize: 512`
- `minzoom: 6`
- `opacity: 0.7`
- insert above road lines and below labels / buildings

### Attribution

When active, add `TomTom` to the scene overlay credit line.

### Files

- `src/overlays/roadTraffic.ts`
- `src/overlays/roadTraffic.test.ts`
- `src/overlays/overlayAnchors.ts`
- `src/main.ts`

## Implementation Order

1. Presets
2. Solar terminator
3. Weather radar
4. Road traffic

Each slice should stay independently shippable.

## Validation Plan

### Unit tests

- solar math sanity checks around equinoxes and solstices with tolerances, not exact-second assertions
- solar polygon ring shape and normalized coordinates
- RainViewer metadata parsing and tile-url construction
- weather refresh logic and status-text formatting
- TomTom key presence logic and tile-url construction

### Manual checks

- `Night` fades away before the zoom-6 mercator switch
- `Weather` disappears above zoom `8`
- the weather layer can be toggled repeatedly without duplicate sources or layers
- dateline-adjacent globe views do not tear the night overlay
- `Road Traffic` shows a disabled but visible toggle when no key is present

## Risks

- RainViewer has no availability guarantees
- RainViewer frame time is only the frame-generation time
- large hemisphere polygons can still show seam artifacts if the ring math is wrong
- OpenFreeMap style ids can change, so the anchor helper must retain fallbacks
- the TomTom slice still adds key friction that the other slices avoid

## Recommendation

Ship presets, night, and weather first. Keep road traffic last and optional because it is the weakest complexity-to-value trade in this set.
