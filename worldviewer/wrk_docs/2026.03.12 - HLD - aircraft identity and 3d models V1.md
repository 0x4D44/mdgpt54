# Aircraft Identity And 3D Models

## Context

Worldviewer already shows live aircraft from OpenSky in the browser. Arthur wants three upgrades:

1. better aircraft identity
2. better aircraft type information
3. a more representative visual, including 3D models at altitude

The design should stay small and should preserve the current GitHub Pages story. That means:

- no new aircraft backend
- no feature flags
- no migration choreography
- no second live aircraft pipeline

## What Matters

- Aircraft must keep working on the static GitHub Pages build.
- The design must stay honest about what the data really tells us.
- The map must stay responsive in a normal browser.
- Each step must be independently useful.

## Non-Goals

- Perfect airline-grade flight identity
- Full photorealistic aircraft simulation
- Per-aircraft animation histories or contrails
- A large aircraft data platform
- A new backend just for aircraft metadata

## Hard Constraints

OpenSky live state vectors give us callsign, position, speed, heading, altitude, and optional aircraft category. They do not directly give us a trustworthy commercial flight number or exact airframe model for every aircraft.

So:

- `callsign` is real and available now
- `flight number` is best-effort only unless another live source is added later
- `aircraft category` can be requested with `extended=1`
- `exact aircraft type` needs a second lookup keyed by `icao24`

Also:

- the existing map pipeline is built around one shared `LiveTrack` shape
- Step 3 must not dump a large 3D library into the main bundle

## Review-Driven Clarifications

The first design review forced four concrete clarifications:

1. `extended=1` must be explicit about field indices
2. the design must choose a single track type strategy
3. Step 2 must name a real metadata source and a bounded static-data plan
4. Step 3 needs explicit bundle, altitude, and LOD rules

This version resolves those points directly.

## Decision Summary

Implement this in three steps:

### Step 1

Enrich the existing OpenSky path with category, better labels, and better 2D aircraft visuals.

### Step 2

Add a static aircraft identity lookup keyed by `icao24` so we can attach type code, registration, manufacturer, and model without adding a new runtime service.

### Step 3

Add a custom 3D aircraft layer that only activates in the camera regimes where it is worth the cost.

## Shared Data Model

Do not introduce a separate `AircraftTrack` pipeline.

Keep one shared `LiveTrack` type and add aircraft-only optional fields to it. Ships will leave those fields unset.

That avoids forking:

- `SnapshotMessage`
- `TrafficClient`
- `tracksToGeoJSON`
- popup rendering
- traffic layer updates

The shape becomes:

```ts
type LiveTrack = {
  id: string;
  kind: "aircraft" | "ship";
  lng: number;
  lat: number;
  heading: number | null;
  speedKnots: number | null;
  altitudeMeters: number | null;
  label: string | null;
  source: "opensky" | "aisstream";
  updatedAt: number;

  // aircraft-only optional fields
  callsign?: string | null;
  flightCode?: string | null;
  aircraftCategory?: number | null;
  geoAltitudeMeters?: number | null;
  aircraftTypeCode?: string | null;
  registration?: string | null;
  manufacturer?: string | null;
  model?: string | null;
  renderModelKey?: string | null;
};
```

## Step 1: Richer Live Aircraft Metadata

### Goal

Make the current aircraft points more informative without changing the architecture.

### Changes

- Request OpenSky states with `extended=1`
- Parse and store:
  - `callsign`
  - `aircraftCategory`
  - `geoAltitudeMeters`
  - existing heading, speed, and barometric altitude
- Replace the current `^` glyph with a small aircraft silhouette set keyed by coarse class
- Add a small callsign formatter for best-effort flight display

### OpenSky Field Map

The implementation must use the documented OpenSky state vector indices explicitly, not by memory.

For the fields this feature needs:

- `0`: `icao24`
- `1`: `callsign`
- `5`: longitude
- `6`: latitude
- `7`: barometric altitude
- `9`: velocity
- `10`: true track
- `13`: geometric altitude
- `17`: category when `extended=1`

This mapping must be kept as constants in the parser.

### Browser Feasibility Note

As of `2026-03-12`, anonymous browser requests to `states/all?extended=1` were manually verified to return 18-field state vectors with CORS enabled.

If OpenSky stops returning category data for anonymous callers later, Step 1 still works:

- callsign stays available
- silhouettes fall back to a generic aircraft symbol

The design does not depend on category being perfect.

### Flight Code Rule

Do not pretend to infer a true commercial flight number.

Only derive `flightCode` when the callsign matches this exact rule:

```ts
/^([A-Z]{3})(\d{1,4}[A-Z]?)$/
```

That is:

- 3-letter ICAO airline prefix
- 1 to 4 digits
- optional trailing suffix letter

If it does not match, show only the raw callsign.

### Rendering

- Low zoom: clusters as today
- Mid zoom: 2D aircraft silhouette instead of `^`
- Popup fields:
  - `flightCode` if present
  - raw `callsign`
  - category label
  - speed
  - altitude
  - age

### Why This First

This step gives visible value immediately and needs no new infrastructure.

## Step 2: Static Aircraft Identity Lookup

### Goal

Attach exact aircraft type information where possible without adding a new runtime dependency.

### Source

Use the public downloadable OpenSky aircraft database CSV as the only metadata source in scope for this step.

Do not scrape third-party aircraft sites.

The README and credits should cite OpenSky for both live traffic and aircraft metadata.

### Refresh Policy

Keep this manual and boring:

- download a recent CSV snapshot when we want to refresh
- run one local build script
- commit the generated compact output

Quarterly refresh is enough for this personal app. We do not need an automated ingestion pipeline.

### Stored Fields

Retain only:

- `icao24`
- `registration`
- `typeCode`
- `manufacturer`
- `model`

Drop everything else during generation.

### Static Artifact Strategy

Use 16 static JSON shards by first hex character:

```text
public/
  aircraft-identity/
    0.json
    1.json
    ...
    e.json
    f.json
```

Why 16:

- much simpler than 256 shards
- smaller and safer than one giant file
- good enough for a personal app

### Runtime Behaviour

- when an aircraft becomes visible, look at the first hex character of `icao24`
- fetch that shard if it is not already in the session cache
- merge identity fields into matching visible aircraft
- if a fetch fails or an identity is missing, keep Step 1 behaviour

The cache is session-scoped only:

```ts
Map<string, Record<string, AircraftIdentity>>
```

No local storage. No persistence.

### Flight Number Handling

Step 2 still does not claim to know a true commercial flight number.

It only improves:

- registration
- type code
- manufacturer
- model

### Rendering Impact

This step unlocks better visual selection:

- `B738` -> 737-family model key
- `A20N` -> A320-family model key
- `B77W` -> 777-family model key
- fallback to category class when type code is missing

## Step 3: 3D Aircraft At Altitude

### Goal

Render aircraft as actual 3D objects at altitude when the camera is close enough for that to matter.

### Rendering Technology

Use a MapLibre custom 3D layer.

Load three.js and the glTF loader dynamically the first time 3D aircraft are actually needed. Do not ship them in the initial application bundle.

### Why Not A Bigger 3D System

We do not need:

- a second renderer
- a persistent aircraft scene graph across the whole world
- a large asset pipeline
- dozens of type-specific models up front

We only need a bounded custom layer that renders a small set of visible aircraft.

### Model Strategy

Start with six class models only:

- `narrow-body`
- `wide-body`
- `regional-jet`
- `bizjet`
- `prop`
- `helicopter`

Do not make Step 3 depend on type-specific `.glb` assets existing.

If later we find license-compatible, low-poly models for common exact types, we can add overrides on top:

- `B738`
- `A320`
- `A21N`
- `B77W`
- `B788`

But class models are the required path. Exact overrides are optional polish.

### Asset Rule

Only use free, redistribution-safe aircraft models. If a suitable model is not available, keep the class fallback and stop there.

### Activation Rules

Use hysteresis so the view does not flicker between 2D and 3D.

Activate 3D only when all are true:

- zoom is at least `9.5`
- pitch is at least `45`
- visible aircraft count is at most `24`

Drop back to 2D when any are true:

- zoom falls below `9`
- pitch falls below `35`
- visible aircraft count rises to `32` or more

This keeps the feature in the camera regime where it will actually read well.

### Altitude Policy

Do not mix altitude sources for 3D placement.

For 3D aircraft positioning:

- use `geoAltitudeMeters` only
- if `geoAltitudeMeters` is missing, keep the aircraft on the 2D path

Barometric altitude can still be shown in popups, but it should not drive 3D placement.

That avoids one aircraft jumping between barometric and geometric height logic.

### Model Key Stability

Once a visible aircraft is assigned a 3D `renderModelKey`, keep that key stable until the aircraft leaves the visible set.

Do not swap from class model to exact type model mid-view when late identity data arrives.

### Positioning

- convert the aircraft position with `MercatorCoordinate.fromLngLat(lngLat, geoAltitudeMeters)`
- rotate by heading / true track
- keep on-ground aircraft on the 2D symbol path

### Performance Budget

Step 3 is only acceptable if it stays cheap enough to matter.

Acceptance target:

- keep interactive frame rate above roughly `45 FPS` in a representative high-pitch aircraft view on Arthur's normal browser machine

If that target is missed:

- lower the cap
- raise the zoom threshold
- reduce model complexity

Do not keep the feature on by default if it clearly harms navigation.

## UI Changes

Extend the aircraft popup to prefer this order:

1. best-effort `flightCode`
2. raw `callsign`
3. registration
4. manufacturer + model
5. type code
6. category label

Do not add a new global toggle for Step 1 or Step 2.

For Step 3, keep automatic activation by camera state. If that proves confusing in practice, a single local `3D Aircraft` checkbox is an acceptable follow-up, but it is not part of the initial design.

## Validation Plan

### Step 1

- unit tests for OpenSky parsing with `extended=1`
- unit tests for callsign-to-flight-code formatting
- manual check that GitHub Pages still shows aircraft

### Step 2

- unit tests for CSV-to-shard generation
- unit tests for shard load and merge logic
- manual check for missing-identity fallback
- manual check that shard fetch failure is non-blocking

### Step 3

- unit tests for model key selection
- manual browser checks for altitude placement, heading rotation, and LOD cutoff
- manual check that 3D assets are lazy-loaded, not included in the initial path

## Risks

- callsign is not the same as a true airline flight number
- OpenSky category data could change or disappear for anonymous callers later
- static identity data will have gaps and stale entries
- even 16 shards can still be uneven by prefix
- 3D aircraft can get expensive fast if thresholds are too loose
- free aircraft model licensing may become the real bottleneck for Step 3

## Recommendation

Proceed exactly in order:

1. richer metadata and 2D silhouettes
2. static `icao24` identity lookup
3. limited 3D aircraft layer

That keeps the design honest, keeps GitHub Pages viable, and avoids turning a personal app into an aviation platform.
