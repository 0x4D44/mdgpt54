# HLD — Traffic Visual Enhancements

## Overview

Four client-side enhancements to the live traffic visualization: altitude-colored aircraft icons, fading trail lines behind aircraft, proper ship silhouette icons with wake trails, and on-demand flight route display from aircraft popups. All four are client-only changes with no new npm dependencies, no server modifications, and no feature flags. Each feature is independently useful and can ship in isolation.

## Features

### 1. Altitude-Colored Aircraft

#### What it does

Colors 2D aircraft icons by altitude band so a glance at the map reveals traffic stratification. Low aircraft appear green, mid-altitude cruise is cyan/blue, and high-altitude traffic is purple. The 3D meshes get the same altitude tint applied to their material color.

#### Design

**No new files.** Changes stay inside existing modules.

**Altitude band mapping (pure function):**

```
altitudeColorExpression(): ExpressionSpecification
```

Returns a MapLibre `"interpolate"` expression over `["get", "geoAltitudeMeters"]` with fallback to `["get", "altitudeMeters"]`:

| Band | Altitude | Color |
|---|---|---|
| Ground / low | 0 – 1 500 m | `#4ade80` (green) |
| Low cruise | 1 500 – 6 000 m | `#facc15` (yellow) |
| Mid cruise | 6 000 – 10 000 m | `#22d3ee` (cyan) |
| High cruise | 10 000 – 12 200 m | `#3b82f6` (blue) |
| FL400+ | > 12 200 m | `#a855f7` (purple) |

Use `"coalesce"` to prefer `geoAltitudeMeters` over `altitudeMeters`, falling back to the default cyan (`#67d0ff`) when neither is present. Ground aircraft (`onGround === true`) get green regardless.

The expression uses `"interpolate", ["linear"]` so transitions between bands are smooth rather than banded.

**2D layer change** — `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts`:

Add `"icon-color"` to the `AIRCRAFT_LAYER` paint block. The aircraft icons are already drawn with `fillStyle: "rgba(103, 208, 255, 0.96)"`, which is opaque cyan. For `icon-color` to tint the icon, the canvas icons must be drawn white (or a neutral base color) so MapLibre's SDF-style tinting works. Change `createAircraftIcon` to draw with white fill (`#ffffff`) and a dark stroke, then let `icon-color` apply the altitude hue at render time. Set `sdf: false` is not needed — MapLibre `icon-color` works on non-SDF images by multiplying against the source pixels, so white source pixels take the full tint.

Alternative: draw icons with `sdf: true` and a white template. Either approach works; white-fill with multiply is simpler and avoids SDF distance-field generation.

**3D material tint** — `C:\language\mdgpt54\worldviewer\src\traffic\aircraft3dLayer.ts`:

Add `altitudeTintColor(altitudeMeters: number): string` (pure function, same band thresholds). In `syncAircraftObjects`, when creating or reusing a mesh instance, set `bodyMaterial.color` to the altitude tint. Since mesh prototypes share materials, clone the material on the instance (`.material = material.clone()`) before mutating. The tint updates each sync cycle, which already runs on every frame that has track changes.

Export `altitudeTintColor` from a shared location so 2D and 3D use the same thresholds.

**Integration points:**

- `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts` — `AIRCRAFT_LAYER` paint, `createAircraftIcon` fill color
- `C:\language\mdgpt54\worldviewer\src\traffic\aircraft3dLayer.ts` — material color in `syncAircraftObjects`
- `C:\language\mdgpt54\worldviewer\src\traffic\trafficHelpers.ts` — add `altitudeTintColor(altitudeMeters: number): string` and `altitudeColorExpression(): ExpressionSpecification`

**No CSS changes.**

#### Test plan

- Unit test `altitudeTintColor` at boundary values: 0 m, 1 500 m, 6 000 m, 10 000 m, 12 200 m, 15 000 m, `null`.
- Unit test `altitudeColorExpression` returns a valid interpolate expression shape.
- Existing `tracksToGeoJSON` tests already confirm `altitudeMeters` and `geoAltitudeMeters` propagate to properties — no change needed.
- Visual smoke test: confirm aircraft visibly change color across altitude bands.

### 2. Aircraft Trail Lines

#### What it does

Draws fading polyline trails behind aircraft showing their recent path over the last 2–5 minutes. Trails are accumulated client-side from snapshot data and rendered as a separate GeoJSON `LineString` layer beneath the aircraft icons.

#### Design

**New file:** `C:\language\mdgpt54\worldviewer\src\traffic\aircraftTrails.ts`

**Key types:**

```ts
type TrailPoint = { lng: number; lat: number; timestamp: number };

type AircraftTrail = {
  id: string;
  points: TrailPoint[];      // ring buffer, oldest first
};
```

**Key functions:**

```ts
/** Max trail points per aircraft. At ~5s snapshot interval, 60 points ≈ 5 minutes. */
const MAX_TRAIL_POINTS = 60;
const TRAIL_MAX_AGE_MS = 5 * 60 * 1000;

/** Update trail store from a new set of aircraft tracks. Returns the updated store. */
function updateTrails(
  store: Map<string, TrailPoint[]>,
  aircraft: LiveTrack[],
  now: number
): Map<string, TrailPoint[]>

/** Remove trails for IDs no longer present in the snapshot. */
function pruneStaleTrails(
  store: Map<string, TrailPoint[]>,
  activeIds: ReadonlySet<string>
): void

/** Convert trail store to a GeoJSON FeatureCollection of LineStrings. */
function trailsToGeoJSON(
  store: Map<string, TrailPoint[]>,
  now: number
): GeoJSON.FeatureCollection
```

**Data flow:**

1. `TrafficClient` publishes `SnapshotMessage` to `main.ts` via `onSnapshot`.
2. In the snapshot callback (already in `main.ts`), call `updateTrails(trailStore, snapshot.aircraft, Date.now())` before `updateTrafficData`.
3. Call `pruneStaleTrails(trailStore, activeAircraftIds)` to evict trails for vanished aircraft.
4. Call `trailsToGeoJSON(trailStore, now)` and set the data on a new GeoJSON source.

**Ring buffer logic** in `updateTrails`: For each aircraft, append `{ lng, lat, timestamp }` if the latest point differs from the tail of the buffer (deduplicate stationary aircraft). If buffer exceeds `MAX_TRAIL_POINTS`, shift the oldest entry. Expire entries older than `TRAIL_MAX_AGE_MS`.

**GeoJSON output:** Each trail becomes a `Feature<LineString>` with properties `{ id, opacity }`. The opacity is based on the trail's freshest point — stale trails fade. Each LineString's coordinate array is ordered oldest-to-newest.

**MapLibre layer** — add to `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts`:

```
Source: "aircraft-trails", type: "geojson", data: EMPTY_FC
Layer: "aircraft-trail-lines", type: "line"
  paint:
    line-color: "#67d0ff"      (or altitude-tinted if feature 1 ships first)
    line-opacity: ["coalesce", ["get", "opacity"], 0.4]
    line-width: ["interpolate", ["linear"], ["zoom"], 5, 0.8, 12, 2.0]
  layout:
    line-cap: "round"
    line-join: "round"
```

Add this layer *before* `AIRCRAFT_LAYER` (so trails render behind icons). Add the source in `addTrafficLayers`. Expose `AIRCRAFT_TRAILS_SOURCE` for the data update call.

**Integration points:**

- `C:\language\mdgpt54\worldviewer\src\traffic\aircraftTrails.ts` — new, pure logic
- `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts` — new source + layer, new `updateTrailData(map, fc)` function
- `C:\language\mdgpt54\worldviewer\src\main.ts` — hold `trailStore`, call update in snapshot callback
- `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts` — `clearTrafficData` also clears trails source

**Memory budget:** `MAX_TRAIL_POINTS` (60) * ~40 bytes per point * worst-case 500 aircraft in viewport = ~1.2 MB. Acceptable.

**No CSS changes.**

#### Test plan

- Unit test `updateTrails`: appending new points, deduplication of stationary aircraft, ring buffer eviction at `MAX_TRAIL_POINTS`, age-based expiry.
- Unit test `pruneStaleTrails`: removal of IDs not in active set, retention of active IDs.
- Unit test `trailsToGeoJSON`: correct LineString geometry, opacity property, empty store returns empty FC.
- Integration: confirm trails render behind aircraft and fade after aircraft disappear.

### 3. Ship Icons & Wake Trails

#### What it does

Replaces the current `">"` text symbol for ships with canvas-drawn ship silhouettes following the same pattern as `createAircraftIcon`. Adds a short tapered wake trail behind each ship proportional to speed. Uses `icon-rotate` for heading instead of `text-rotate`.

#### Design

**New file:** `C:\language\mdgpt54\worldviewer\src\traffic\shipIcons.ts`

**Key constants:**

```ts
const SHIP_ICON_SIZE = 48;
const SHIP_ICON_PIXEL_RATIO = 2;
const SHIP_ICON_NAME = "ship-generic";
```

**Key function:**

```ts
/** Draw a top-down ship silhouette on a canvas, returning ImageData for MapLibre. */
function createShipIcon(): ImageData | { width: number; height: number; data: Uint8ClampedArray }
```

The silhouette is a pointed-bow, flat-stern hull shape drawn with `context.fill()` and `context.stroke()`, using the existing `drawClosedShape` pattern from `trafficLayers.ts`. Fill color: white (`#ffffff`) so `icon-color` can tint it with `--gold` (`#f4c989`). Stroke: dark outline.

The hull shape: a simple 8-point polygon approximating a cargo/tanker outline in profile (pointed bow at top, parallel sides, flat stern). A single generic shape — vessel type differentiation is deferred unless the AIS data reliably includes vessel type codes (it currently does not propagate through `LiveTrack`).

**Wake trail:** A second canvas icon (`ship-wake`) drawn as a tapered V-shape in semi-transparent white, rendered as a separate symbol layer behind the ship layer. The wake length scales with `speedKnots` via an `icon-size` expression:

```ts
function shipWakeSizeExpression(): ExpressionSpecification
// ["interpolate", ["linear"], ["get", "speedKnots"], 0, 0, 5, 0.3, 15, 0.6]
```

Ships with `speedKnots === 0` or `null` get no wake (size 0).

**Layer changes** — `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts`:

Replace the `SHIPS_LAYER` from `type: "symbol"` with text to `type: "symbol"` with icon:

```
  icon-pitch-alignment: "map"
  icon-keep-upright: false

  icon-color: "#f4c989"

```

Add a wake layer (`SHIPS_WAKE_LAYER`) before `SHIPS_LAYER`:

```

  icon-image: "ship-wake"
  icon-size: shipWakeSizeExpression()
  icon-allow-overlap: true
  icon-ignore-placement: true
  icon-rotate: ["coalesce", ["get", "heading"], 0]
  icon-rotation-alignment: "map"
  icon-anchor: "top"
paint:
  icon-opacity: ["*", ["coalesce", ["get", "opacity"], 1], 0.4]
```

Call `ensureShipIcons(map)` in `addTrafficLayers` alongside `ensureAircraftIcons(map)`.

### 4. Flight Route Display

#### What it does

When the user clicks an aircraft popup, a "Show route" action fetches the flight route (origin and destination airports) and draws a great-circle arc on the map. The arc dismisses when the popup closes.

#### Design

**New file:** `C:\language\mdgpt54\worldviewer\src\traffic\flightRoute.ts`

**Static data file:** `C:\language\mdgpt54\worldviewer\src\traffic\airportCoords.json`

A JSON object keyed by ICAO code with `[lat, lng]` values for the top ~3 000 airports worldwide. Generated offline from OurAirports CSV. Approximately 60 KB gzipped. Imported as a static module so it is bundled and tree-shaken normally.

**Key types:**

```ts
type AirportCoord = { icao: string; lat: number; lng: number };
type FlightRouteResult = {
  origin: AirportCoord | null;
  destination: AirportCoord | null;
  arc: [number, number][];       // [lng, lat] coordinate pairs for the great-circle arc
} | null;
```

**Key functions:**

```ts
/** Lookup airport coordinates by ICAO code. Returns null if not in dataset. */
function lookupAirport(icao: string): AirportCoord | null

/** Fetch route from OpenSky API. Returns origin/destination ICAO codes or null. */
async function fetchFlightRoute(
  callsign: string,
  signal?: AbortSignal
): Promise<{ originIcao: string; destinationIcao: string } | null>

/** Compute great-circle arc coordinates between two points.
 *  Returns an array of [lng, lat] pairs with `steps` intermediate points. */
function greatCircleArc(
  from: [number, number],
  to: [number, number],
  steps?: number            // default 64
): [number, number][]

/** Full pipeline: fetch route, resolve airports, compute arc. */
async function resolveFlightRoute(
  callsign: string,
  signal?: AbortSignal
): Promise<FlightRouteResult>
```

**Great-circle arc math** (pure function, no dependencies): Spherical interpolation (slerp) between two lat/lng points, emitting `steps` intermediate coordinates. This is ~20 lines of trigonometry. No library needed.

**OpenSky route API:** `GET https://opensky-network.org/api/routes?callsign={CALLSIGN}`. Returns `{ "callsign": "...", "route": ["EGLL", "KJFK"] }`. The API is unauthenticated and rate-limited. Handle 404 (no route found) and network errors gracefully — return `null`, do not retry.

**MapLibre rendering** — `C:\language\mdgpt54\worldviewer\src\traffic\trafficLayers.ts`:

Add a source and layer for the route arc:

```

  paint:
    line-color: "#a78bfa"      (light purple, distinct from traffic)
    line-width: 2
    line-dasharray: [4, 3]
    line-opacity: 0.7
  layout:
    line-cap: "round"
    line-join: "round"
```

**New functions in trafficLayers.ts:**

```ts

function clearFlightRoute(map: Map): void
```

`showFlightRoute` sets the `flight-route` source data to a single `Feature<LineString>`. `clearFlightRoute` resets it to `EMPTY_FC`.

**Popup integration** — modify `showTrafficPopup` in `trafficLayers.ts`:

For aircraft popups where `callsign` is present, add a clickable "Show route" text element to the popup HTML. Wire a click handler on the popup DOM element (using `popup.getElement().querySelector(...)`) that:

1. Calls `resolveFlightRoute(callsign, signal)`.
2. If successful, calls `showFlightRoute(map, result.arc)`.
3. Updates the popup HTML to show origin/destination ICAO codes.
4. If the lookup fails, shows "Route unavailable" in the popup.

Wire `clearFlightRoute(map)` into `clearTrafficPopup` so the arc disappears when the popup closes or when the user clicks elsewhere.

**Abort handling:** Create an `AbortController` when the "Show route" button is clicked. Abort it in `clearTrafficPopup` to cancel in-flight requests when the popup is dismissed before the fetch completes.

## Implementation Order

1. **Altitude-Colored Aircraft** — smallest change, modifies only existing files, no new modules. Provides immediate visual value. Unblocks trail coloring in step 2.
2. **Ship Icons & Wake Trails** — self-contained icon replacement. No dependency on other features. Improves ship rendering quality independently.
3. **Aircraft Trail Lines** — requires the snapshot callback wiring in `main.ts`. Benefits from altitude colors (optional: color trails by altitude). Larger surface area but isolated pure logic.
4. **Flight Route Display** — largest feature, requires new static data file, fetch integration, popup DOM manipulation, and abort handling. Most testing complexity. Ship last so the simpler features are stable first.

## Open Questions

1. **Icon tinting approach:** Should `createAircraftIcon` draw white-fill icons and rely on MapLibre `icon-color` multiplication, or should we switch to SDF icons for true tinting? White-fill multiplication is simpler but may produce slightly washed-out dark strokes. Need a quick visual test.
2. **3D material cloning cost:** Cloning `MeshStandardMaterial` per aircraft per sync cycle could be expensive with many visible 3D aircraft. Should we pool materials by altitude band (5 materials) instead of per-aircraft? Likely yes — 5 shared materials is cleaner.
3. **Trail color by altitude:** If feature 1 ships first, should trail lines inherit the altitude color of the aircraft, or stay a uniform cyan? Altitude-colored trails add visual richness but make the expression more complex (need to propagate altitude into trail LineString properties). Suggest uniform cyan for V1, revisit if it looks flat.
4. **Airport data freshness:** The static `airportCoords.json` will go stale over time as airports open/close. Is a manual annual refresh acceptable, or should we fetch from a CDN? Static is simpler and matches the no-new-dependencies constraint.
5. **OpenSky route API reliability:** The unauthenticated OpenSky route endpoint may be rate-limited or unavailable. Should we cache successful lookups in a session-scoped `Map<string, FlightRouteResult>` to avoid redundant fetches? Probably yes — add a simple in-memory cache keyed by callsign.
6. **Ship vessel type differentiation:** The current `LiveTrack` type does not carry vessel type codes from AIS data. If the relay starts providing vessel type, should we add hull shape variants (tanker, cargo, passenger, fishing)? Defer until data is available — use a single generic hull for now.
