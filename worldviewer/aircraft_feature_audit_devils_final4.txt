Based on my full read of the audit packet, here is the devil's advocate review.

---

## Summary

The implementation spans three files (openskyDirect.ts, aircraftIdentity + identityData, aircraft3d + aircraft3dLayer) plus a generation script. All three HLD steps are present. The HLD-specified shape, thresholds, shard scheme, and popup order are replicated accurately. No file was rewritten or forked against the HLD's explicit instruction to preserve the single `LiveTrack` pipeline.

---

## Critical Gaps

**None confirmed.** One near-gap is worth recording:

**`updatedAt` uses local poll time, not OpenSky `time_position`**

`openskyDirect.ts:60` — `updatedAt: now` — sets freshness to the local wall-clock at the moment of the poll, ignoring OpenSky's `time_position` at state-vector index 3.

The HLD's field map (HLD lines 132–144) does **not** list index 3 (`time_position`) or index 4 (`last_contact`). The HLD is therefore silent on which timestamp to use for `updatedAt`. However, the HLD explicitly requires the design to stay "honest about what the data really tells us" (HLD line 21). With a 15-second poll interval (`OPENSKY_POLL_MS = 15_000`), a position that OpenSky last updated 55 seconds ago will arrive and be displayed as "0s ago" — then stale after the 60-second threshold with no intermediate warning. This is a correctness gap in the age display, not a plan violation, because the HLD never specified which source timestamp to use. It is worth a deliberate decision either way.

---

## Assumptions To Challenge

**1. Staleness epoch (above).** The system assumes `updatedAt = Date.now()` at poll time is an acceptable proxy for "how old is this position." OpenSky's crowdsourced feed can hold positions that haven't moved for tens of seconds. The stale-threshold logic (`STALE_THRESHOLD_MS = 60_000`) is internally consistent with poll intervals but overstates recency to the user. If the HLD's honesty constraint is interpreted strictly, this should use `time_position` or at least clamp displayed age against the OpenSky-reported timestamp.

**2. Three.js lazy-load contract is injectable but not directly verified in the packet.** `aircraft3dLayer.ts` exposes `Aircraft3dController` with an injectable `loadThree` parameter (evidenced in `aircraft3dLayer.test.ts:87-88`). Production construction in `main.ts:884` omits `loadThree` entirely, relying on the default. If the default is not a dynamic `import("three")`, three.js lands in the initial bundle — violating HLD line 311. The packet does not include the full `aircraft3dLayer.ts` body so the default's implementation cannot be directly confirmed here. The procedural mesh functions all receive `THREE` as a parameter (correct lazy-load pattern), but the default constructor argument must be confirmed to be `() => import("three")` or equivalent.

**3. Default 3D class is `narrow-body` for all unresolvable aircraft.** `aircraft3d.ts:237` returns `"narrow-body"` when no prefix, keyword, or category matches. At high-density departure zones (EGLL, EDDM), where many squawking aircraft lack Step-2 identity, the overwhelming majority of 3D objects will be indistinguishable narrow-body shapes. This is within the HLD's intent ("procedural class meshes are intentionally approximate") but is worth knowing.

**4. `resolveAircraft3dModeFromVisibleCount` — ON condition uses `<=` not `<`.** `aircraft3d.ts:123` activates when `visibleRenderableCount <= AIRCRAFT_3D_ON_COUNT` (i.e., 24 is still ON). The HLD says "at most 24" (HLD line 369), so `<= 24` is a faithful implementation. No gap — just confirming the boundary is intentional.

**5. Shard tuple format is undocumented in HLD.** Shards are compact tuples `[registration, typeCode, manufacturer, model]` instead of named-key objects. The HLD only says "compact output"; the tuple format is not specified. The `parseAircraftIdentityShard` function validates and expands tuples on the client. This is safe but is an implicit serialisation contract between the generator script and the runtime — any manual editing of shard files will be non-obvious.

---

## Verdict

**PASS — no critical plan-vs-implementation gaps.** All HLD-specified thresholds, field indices, regex rules, shard count, class model set, popup ordering, lazy-load architecture, and single-pipeline constraint are present and match the documented design. The `updatedAt = now` choice is the only finding that could matter to users, and it stems from an HLD omission rather than a deviation. Recommend an explicit decision on whether to use OpenSky's `time_position` before the next visible staleness complaint.
