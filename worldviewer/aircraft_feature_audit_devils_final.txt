## Summary

The audit packet includes the HLD, all six implementation files (trafficTypes, openskyDirect, aircraftIdentityData, aircraftIdentity, aircraft3d, aircraft3dLayer), the generation script, and their test suites. I read the full packet against the HLD. The implementation is substantially faithful to the design; only one genuine shape deviation was found.

---

## Critical Gaps

**1. `onGround` field added to `LiveTrack` without appearing in the HLD's explicit shared data model**

The HLD's *Shared Data Model* section (lines 82–110) provides an explicit TypeScript type definition for `LiveTrack`. That definition does not include `onGround`. The implementation adds `onGround?: boolean | null` at `trafficTypes.ts:17`.

This field is load-bearing for Step 3: `getAircraft3dAltitudeMeters()` (`aircraft3d.ts:160`) returns `null` if `onGround === true`, which is how the HLD's requirement "keep on-ground aircraft on the 2D symbol path" (`HLD:389`) is satisfied. Without this field the Step 3 activation logic is broken. But the HLD never updated its stated type shape to include it, leaving the data model section incorrect as written.

This is a silent extension to the shared canonical type that was never ratified in the design document.

---

**No other critical gaps were found.** All three steps follow the HLD on every named requirement: field indices, regex, shard count, generation guardrails, hysteresis thresholds (10.5/10, 45/35, 24/32), lazy Three.js load, six class meshes, `MercatorCoordinate` positioning, popup field priority order, and the session-only cache.

## Assumptions To Challenge

**A1. "Visible unclustered aircraft count" vs. "renderable airborne count"**

HLD (`lines 376–377`) says the activation count should be "the raw aircraft track array filtered by viewport bounds." The implementation counts via `buildRenderableAircraft3dTracks()` (`aircraft3d.ts:128–155`), which additionally excludes grounded aircraft and aircraft with no altitude. At a busy airport with 40 grounded aircraft and 3 in the air, the HLD count would be 43 (no 3D); the implementation count would be 3 (3D activates). The implementation's behavior is arguably correct—grounded aircraft never need 3D rendering—but it diverges from the letter of the HLD's criterion without acknowledgment.

**A2. `updatedAt` uses browser poll time, not aircraft last-contact time**

`parseOpenSkyStates()` (`openskyDirect.ts:60`) sets `updatedAt: now` where `now = Date.now()`. OpenSky state vectors carry `time_position` at index 3 and `last_contact` at index 4, neither of which is consumed. The popup "age" row and the stale-detection (`isTrackStale`, `trackOpacity`) therefore reflect time-since-browser-poll, not time-since-aircraft-transponder-contact. For OPENSKY_POLL_MS=15 000 ms the maximum error is one poll cycle, so the practical impact is small—but the design document says nothing about this choice, leaving it as an unstated assumption.

**A3. URL resolution complexity not implied by the HLD**

The HLD says the shard scheme is "still static-host friendly for a personal app" (line 234). `buildAircraftIdentityShardUrlFromBase()` (`aircraftIdentity.ts:111–119`) implements non-trivial resolution logic: relative vs. absolute base URL detection, trailing-slash normalisation, HTML document path detection, fallback to `document.baseURI` vs. `location.href` vs. `location.origin`. Eleven test cases cover this (`aircraftIdentity.test.ts:186–233`). The HLD gives no hint this complexity was anticipated. If the logic is wrong for an edge case on a future hosting change, it will not be obvious why.

**A4. Prototype geometry/material sharing on dispose**

The comment at `aircraft3dLayer.ts:3773` notes that `clone(true)` shares geometry and material references between the prototype and scene instances. `disposePrototypes()` (`aircraft3dLayer.ts:4028–4050`) iterates prototype meshes and calls `.dispose()` on their geometries and materials. Because those same objects are referenced by scene instances, disposing the prototype immediately invalidates all live scene instances. The code is safe only because `dispose()` removes all objects from the scene before calling `disposePrototypes()`. This ordering dependency is not documented and is one line of change away from a GPU resource use-after-dispose.

---

## Verdict

**No blocking gaps.** The implementation delivers all three steps as designed. The one genuine plan-vs-implementation deviation—`onGround` missing from the HLD's stated `LiveTrack` shape—is functionally necessary and correctly implemented; it is a documentation gap in the HLD, not a code defect. The four assumptions flagged above are real unstated design decisions but none of them causes incorrect behaviour in the current configuration.
